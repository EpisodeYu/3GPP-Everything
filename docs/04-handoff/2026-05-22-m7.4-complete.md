## 3. 关键设计决策

### 3.1 ContextVar 而非 AgentState 携带 user_id

**为什么**：CLAUDE.md §3 surgical changes 原则。AgentState 已有 14+ 字段，加 `user_id`
会牵动 LangGraph PostgresSaver checkpoint schema、所有节点签名、agent 测试
fixture。ContextVar 的好处：

- FastAPI 单 request = 单 asyncio Task，ContextVar 内部传递正确（`asyncio.create_task`
  自动 `copy_context()`）
- LiteLLMClient / web_search_tool 不需要 caller 显式传 user_id
- ingestion / eval / 后台 job 这种"无 user"场景天然 skip

**风险**：multi-worker uvicorn 部署下若把 LLM 调用挪到 worker 子进程会断链。M4 单
进程 dev 不受影响；M8 上线前如要切 worker 模式需重审。

### 3.2 跨方言 upsert：UPDATE-then-INSERT-on-IntegrityError

**为什么**：项目 PG（生产）+ SQLite（集成测）双方言。`INSERT ... ON CONFLICT
DO UPDATE` 两边语法都支持但 SQLAlchemy 写法分叉（`postgresql.dialects.insert` /
`sqlite.dialects.insert`），且不能跨方言复用。

**做法**：先用 `UPDATE` 自身列引用做 atomic 累加；rowcount=0 退化 INSERT；INSERT
撞 UNIQUE → rollback 后再 UPDATE 一次。三步全部跨方言可用，并发 race 也兜得住。

### 3.3 fire-and-forget hook，不阻塞业务路径

**为什么**：LLM 调用本来就慢（500ms-3s），usage 写库再额外阻塞 5-20ms 是无谓延迟。
而且 PG 偶尔抖动时 chat 不该跟着失败。

**做法**：`asyncio.ensure_future(record_*_usage(...))` 调度成新 Task；hook 内部
异常 swallow + log.warning。代价：测试断言时需要 `await asyncio.gather(*pending)`
等 fire-and-forget Task 完成（已在 `test_chat_response_triggers_record_llm_usage` 处理）。

### 3.4 Voyage rerank token 估算 fallback

**为什么**：LiteLLM 透传 voyage rerank 的 `usage` 字段口径不统一（有时回 cohere
形式 `meta.tokens.input_tokens`，有时纯空），客户端按 `len(text)/4` 估算更稳定。
M7.4 Q2 决策仅 log warning，不依赖精确数字 → 估算可接受。

未来如要精确：在 `record_rerank_usage` 加 raw payload 透传 + 改 `pricing.py::rerank_billable_tokens`
读 voyage 真实回包字段。

### 3.5 alerts 用 apscheduler `AsyncIOScheduler` 而非 cron / Celery

**为什么**：M7.4 仅 log warning，开销极低；起独立 cron 容器或 Celery beat 是
overkill。`AsyncIOScheduler` 进程内挂在 FastAPI lifespan 上，零额外依赖。

**风险**：multi-worker 部署下 N 个 worker 各跑一次 daily job → log 行被打 N 次。
M4 单进程 dev 没问题；M8 上线时如启用 `--workers 4` 需在外层做 leader 选举（Redis
SETNX 或文件锁），或干脆把 scheduler 拆成独立进程容器。

### 3.6 阈值 ≤ 0 视作 disabled，不抛错

**为什么**：用户可能想关单档（比如只要 critical 不要普通 warn），用 0 比再加个
`ALERT_DAILY_ENABLED=false` 字段简洁。

## 8. 自主决策记录（按 CLAUDE.md §4.3）

1. **ContextVar 内部 key 名 `tgpp_current_user_id`**：避免与 langchain / langfuse
   可能的 ContextVar 冲突。
2. **`_approx_token_count(text) = len/4`**：OpenAI 公开口径下限近似；中文 1 字 ≈
   1 token 比英文密集 → 4 字符是保守低估，对计费保守安全。M7.4 Q2 仅 log warning，
   精度足够。
3. **`ALERT_DAILY_AGGREGATE_HOUR=1`**：与 daily eval 02:00 不重叠；本地时区 +1h
   足够保留 daily 写完到聚合的间隔。
4. **alerts 聚合"昨日"而非"今日"**：今日仍在累加，跑半路读会得到不完整数字
   → 总是看昨日 final 数。月累计仍包含今日（`day <= today`），看趋势用。
5. **未知模型走 `_UNKNOWN_*` 单例 cost=0**：避免 hook 因新增模型未及时上表抛错；
   token 仍写入 ApiUsage 列让运维看见"有未知模型"。
6. **`pricing.py` 用 dataclass 而非 dict**：类型安全 + IDE 跳转好；测试 import
   `get_*_price()` 拿到 dataclass 后 `.billed` 字段做断言更稳。
7. **不动 `ApiUsage.rerank_calls` 改 `rerank_tokens`**：保持 schema 不变；token
   数走 structlog DEBUG，未来真要拆列再 alembic（M4.10 已有迁移历史）。
8. **`web_search_tool` hook 写在成功返回前一行**：失败路径（缺 key / tavily 错）
   不计费，只成功 search 才记一次；未来加 search_depth=advanced 折扣可再扩
   `WebSearchPrice`。
