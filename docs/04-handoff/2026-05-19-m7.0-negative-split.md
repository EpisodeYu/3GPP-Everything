## 2. 两个子类的语义

### 2.1 `negative_no_steelman` — 无理可推的伪问题

特征：问题本身**没有合理重写**，agent 该明示规范未涉及。

示例：
- "TS 38.331 是否定义了 6G 卫星 NR 的轨道修正 delta-v？"
- "5G 协议规定的 UE 抗辐射等级是多少？"
- "R20 是否冻结了 6G NR？"

期望回答风格："3GPP 当前规范未涉及 / 未定义 X。"（中英按题目 language 切词表，见 §4）

### 2.2 `negative_clarify` — 有可救概念的错问

特征：问题里有**明显概念混淆**或**术语误用**，agent 应反问或在高置信度下猜真实问题。

示例：
- "5G UE 的 MAC 地址格式是什么？"（PDU Session 是 L3，无 MAC；真问题可能是 IP 地址或 C-RNTI）
- "5G AKA 是否支持 GBA？"（GBA 是 Pre-5G 的概念；真问题可能是 5G AKA 与 AKMA 的关系）
- "5G NR 的 DCI format 0_2 的 OFDM 子载波间距是多少？"（DCI format 不决定 SCS；真问题可能是 BWP 或 numerology）

期望回答风格：

- **优先**：明确指出概念混淆 + 反问"您是不是想问 X？"
- **可选**：高置信度时直接给出推断版答案，但**必须带免责**（"如果您问的是 X，则..."）
- **禁止**：默认走"未找到"短语（这会让用户以为是工具问题）

## 3. Schema 改动（`eval/validators/golden.py`）

### 3.1 CATEGORY_ENUM 更新

```python
CATEGORY_ENUM = frozenset({
    "definition", "procedure", "multi_section",
    "table_lookup", "formula",
    "negative_no_steelman", "negative_clarify",  # 替换原 "negative"
})
```

### 3.2 字段约束对比表

| 字段 | `negative_no_steelman` | `negative_clarify` |
|---|---|---|
| `expected_specs` | 必空 `[]` | 可空 / 可非空 |
| `expected_facts` | 必空 `[]` | 可空 / 可非空 |
| `must_say_not_found` | 必填 `true` | 必填 `false` |
| `expected_clarification_topic` | 必空 `[]` 或省略 | **必填，≥ 1 条** |
| `forbidden` | 必填且非空 | 必填且非空 |
| `notes` | 必填（"为什么这是无 steelman 的"） | 必填（"用户可能真正想问什么"） |

`expected_specs` 在 clarify 类**非空**的语义：agent 不仅要 clarify，还应该引这个 spec 给出推断答案；非空时 retrieval 指标按常规规则评。

### 3.3 新字段：`expected_clarification_topic`

- 类型：`list[str]`
- 语义：clarify 类必须命中其中**至少 1 条**，由 LLM judge 判定（substring + 语义双向，见 §5）
- 写法：用名词短语，不要长句。例：`["PDU Session 的 IP 地址", "C-RNTI", "RNTI 类型"]`

### 3.4 新增校验规则

- `negative_no_steelman` 题填了 `expected_clarification_topic` 非空 → error `wrong_field_for_no_steelman`
- `negative_clarify` 题缺 `expected_clarification_topic` 或为空 → error `negative_clarify_missing_topic`
- `negative_clarify` 题 `must_say_not_found=true` → error `negative_clarify_must_not_be_not_found`
- `negative_clarify` 题 `forbidden` 为空 → error `negative_clarify_forbidden_required`（防止漏防御幻觉）
- 既往规则（`negative_*` 必须 `forbidden` 非空、`expected_facts` 与 `expected_specs` 一致性等）保留

## 4. 评测断言（M7.1 runner，`eval/runner.py`）

### 4.1 `negative_no_steelman`

沿用 `must_say_not_found_passed`：按题目 `language` 字段切中英短语词表。
- en：`not found` / `not specified` / `no such` / `does not define` / `is not defined in` / `outside the scope`
- zh：`未找到` / `未定义` / `规范未规定` / `不涉及` / `不在范围内` / `没有相关规定`

通过 = `must_say_not_found_phrase_matched AND forbidden_violations == []`

### 4.2 `negative_clarify`

新加 `clarification_judge_passed`：

```python
# eval/judges/clarification.py（建议新建独立文件，方便单测 + mock）
async def judge_clarification(
    question: str,
    answer: str,
    expected_topics: list[str],
    forbidden: list[str],
    language: str,
) -> ClarifyJudgement: ...
```

Judge LLM prompt（中英双语模板，按 `language` 切）输出 JSON：

```json
{
  "clarified_premise": true,            // 是否明示问题前提有误
  "touched_topic": ["..."],             // 命中的 expected_topics 子集
  "inferred_and_answered": false,       // 是否在高置信度下推断真实问题并作答
  "hallucinated_forbidden": false       // 是否触及 forbidden（双 check：runner 已先做 substring 扫，这里是语义补刀）
}
```

通过条件：

```
(clarified_premise OR inferred_and_answered)
  AND touched_topic 非空
  AND NOT hallucinated_forbidden
  AND forbidden_violations == []        # runner substring 扫的硬指标，独立判
```

### 4.3 `EvalResult` dataclass 增字段

```python
@dataclass
class EvalResult:
    ...
    must_say_not_found_passed: bool | None      # 只 no_steelman 有值
    clarification_judge_passed: bool | None     # 只 clarify 有值
    clarification_judge_raw: dict | None        # judge 原始输出，debug 用
```
