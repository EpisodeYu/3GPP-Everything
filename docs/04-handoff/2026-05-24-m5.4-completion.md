### 关键设计点

#### 1. ChatController 状态机扩展

```
[idle] --send--> [streaming] --pause--> [paused] --resume--> [streaming]
                     |                     |
                     +--cancel--> [cancelling] --(cancelled event)--> [cancelled]
                     |
                     +--final--> [done] --end--> [idle/flush]
                     |
                     +--error--> [error]
```

- 新增 `RunStatus.paused`：保留 `partialAnswer / nodes / chunksHit / runId` 等 run 状态，
  让 UI 在 paused 期间仍能展示流式上下文（用户暂停就是为了"先看看"）
- `_isResume` 内部标记：resume 路径走 PG refetch 而非本地拼装 user+assistant，避免
  - stub 重复：后端在 `send` 期已 INSERT 空 assistant；paused 关流后 stub 仍在 PG，
    `_loadHistoryFromPg` 过滤掉这个 stub；
  - 内容空：resume final 时后端 UPDATE 了 stub 的 content，PG 一刷就回到正确状态
- `_onStreamDone` 在 `paused` 态下静默退出，不再 `_markError('stream_closed')`

#### 2. Composer 双按钮形态

```
idle      → [发送]
streaming → [暂停 (outline)] [取消 (filled)]
paused    → [恢复 (filled)] [取消 (outline)]
```

- 暂停在 streaming 中是 OutlinedButton（次要），取消是 FilledButton（主要）—— 用户能"快速跑完"是默认期待，暂停是次选
- paused 中颠倒：恢复是 FilledButton（主要），取消变成 outline 副选
- `onPause / onResume` 用 nullable 回调：调用方不提供时按 backward-compat 不显示，避免老代码爆掉
- TextField `enabled: !isRunning && !isPaused`，hint 切换 "问点什么 / 正在生成… / 会话已暂停，点恢复继续"

#### 3. archived_branch 视觉与"回到主线"

- composer 整体不渲染，替换成 `_ArchivedBanner`：浅灰背景 + fork 图标 + 文案
  "这是从主线 fork 出的历史分支，只读。" + "回到主线" 按钮（仅当 `forkedFromSessionId` 非空）
- _MessagesList 在 archived 下不挂 GestureDetector → 长按消息不弹菜单（避免在历史分支上又 fork 出新分支的混乱）
- Header 的会话设置菜单（删除最后 N 轮）在 archived 下隐藏

#### 4. 长按菜单设计

- assistant：6 项（复制全文 / 点赞 / 点踩 / 收藏 / 添加笔记 / 详细反馈），用 `showModalBottomSheet`
- user：2 项（复制 / 从这里重问），用 `showModalBottomSheet`
- 内容用 `SingleChildScrollView` 包裹避免小屏 RenderFlex 溢出
- `MessageBubble` 的 `MarkdownBody.selectable=false`：让父级 `GestureDetector(behavior: opaque)`
  在 gesture arena 里赢下 LongPressGestureRecognizer，否则 markdown 内置 selectable 会抢走

#### 5. 引用 fork 的 checkpoint 选择

M5.4 MVP：`/sessions/{sid}/fork` 调用前先 `GET /sessions/{sid}/checkpoints` 取最近一条
checkpoint（按时间倒序的 items.first）作为 `checkpoint_id`。

**已知限制**：UX 上"从这里重问"对任意 user 消息都展示，但实际 fork 点固定是
最近 checkpoint。等价语义是"从 **当前会话最尾部** 分叉"，不是"从这条消息之前的状态分叉"。
原因：`MessageOut` 没暴露 `langgraph_checkpoint_id` 字段。要做精确语义需要后端
schema 改动（M5.5 / M5.6 可加），M5.4 不动后端。

实践上：用户最常用的场景是"最后一条回答不满意，换个问法"，最近 checkpoint
正好就是那个语义点，95% 用例可行。

### sse_client cast 修复（顺带）

测试中暴露：`Stream<Uint8List>.transform(utf8.decoder)` 在 Dart 3.x runtime 上
触发 `Utf8Decoder is not a subtype of StreamTransformer<Uint8List, String>` 错误。
原因是 dio 的 `ResponseBody.stream` 是 `Stream<Uint8List>`，`utf8.decoder` 是
`StreamTransformer<List<int>, String>`，runtime contravariant 检查失败。

修复：`sseFramesFromBytes` 内显式 `bytes.cast<List<int>>()` 拿到 element 类型固定
为 `List<int>` 的包装流再 transform。M5.2 期间没暴露这个 bug 是因为
`FakeMessagesApi` 走 in-memory `Stream<ChatEvent>` 完全绕过了 bytes 解析路径；
M5.4 给 `checkpoint_api.dart` 写了 scripted dio adapter 测试 resume SSE 流才命中。

**生产影响**：M5.2 既有功能（send → token 流）在真实 dio + 浏览器环境下也走这条路。
M5.2 web-smoke 没报错说明在 chrome / Linux 上 transform runtime 检查可能更宽松，
但 cast 是更鲁棒的写法。M5.4 后 web-smoke 仍 `All tests passed`，回归过关。

## 自主决策记录（CLAUDE.md §4.3）

1. **`MarkdownBody.selectable: false`**：原本 selectable=true 让用户能选文本，
   但 selectable 自带 LongPressGestureRecognizer 与父级 GestureDetector 在 arena 里
   竞争，导致测试 longPress 不触发。改 false 牺牲文本选择，换回"长按 → 菜单 → 复制全文"
   的统一路径。生产体验下用户复制走菜单更直观一些（一次长按 → 一次点击 vs.
   长按 → 拖选 → 复制按钮）。如果 UX 反馈强烈想要文本选择，M5.6 polish 可加
   独立"选择模式" toggle。

2. **sse_client `.cast<List<int>>()`**：与原始 transform 等价但通过运行时检查。
   不改 API 签名，零行为变化。注释里写明触发条件 + 为什么需要。

3. **Composer 双按钮顺序**：streaming → 暂停在左 / 取消在右；paused → 恢复在左 / 取消在右。
   "主操作在左 / 取消在右" 一致，避免视觉跳跃。

4. **pause API 失败 → 回退 streaming + errorMessage**：如果后端 pause 路由 500 了，
   不能让 UI 卡在 paused 但实际后端没暂停的状态；回到 streaming 让用户再试或直接取消。
   生产期 pause 失败极罕见。

5. **resume 走 PG refetch 而非本地拼接**：resume 的 final 来到时，后端已 UPDATE
   stub assistant 的 content；前端拉一次 history 比"本地拼半个 assistant + 不知道
   user 之前的输入是什么"靠谱得多。代价是一次额外 round-trip（一两百毫秒），
   能接受。

6. **rollback 与跑中 run 冲突在 controller 层抛 StateError**：让 UI 自己决定怎么提示
   用户（snackbar / dialog / 禁用入口）。chat_page 的当前实现：rollback 入口在
   header 设置菜单，不区分是否在跑（用户可能跑中也想清理），异常路径上 SnackBar
   "回滚失败：rollback_with_inflight_run"。也行。

7. **fork 默认用最近 checkpoint**：UX 简化 + 后端 schema 不动。见上"留给人审"
   节的限制说明。

8. **archived_branch 不再弹长按菜单**：避免"在分叉历史里又分叉"造成多级 archived
   会话嵌套，UX 混乱。如果将来要支持"从历史分支再分叉"，再加一层 `forked_from_branch_id`
   或类似字段。

9. **删除最后 N 轮 slider 1-10**：与后端 `RollbackBody.last_n` 的 ge=1, le=500 区间
   保持 UX 友好；500 这种数字对用户没意义，超过 10 应该走"清空会话"或"删除会话"。

10. **长按菜单用 bottom sheet 而非 PopupMenu / Dialog**：bottom sheet 在移动端
    更顺手（拇指能轻松够到底部），桌面端也美观。下方有 drag handle 提示可关闭。

11. **`FakeMessagesApi.history` 改成可变字段**：测试需要模拟"调 rollback 后 PG
    返回不同 history"。原本 `final List<MessageOut>` 限制了这点。改成可变字段不影响
    生产代码（fake 只用于测试）。

12. **chat_page_m54_test viewport 升到 1000**：6 个长按菜单项加 drag handle 在
    默认 800x600 viewport 下溢出。仅在该测试 file 局部 resize，避免影响其他文件。

13. **Clipboard.setData fire-and-forget**：测试环境的 platform channel mock 处理
    setData 有时不响应；用 `.catchError` 不阻塞 UI 提示，让 SnackBar 一定能显示。

## 剩余风险 / 已知问题

- **fork 点固定为最近 checkpoint**（见"留给人审"第 3 项）：UX 上"从这里重问"对任意
  user 消息都展示，实际 fork 自最近 checkpoint。需后端给 message 加 checkpoint_id
  字段才能精确定位。

- **paused 会话的清理策略**：用户长期不 resume，session.status='paused' 留在 DB +
  LangGraph checkpoint 占空间。`docs/03-development/04-backend-api.md §11` 风险表
  已有"后台任务每天清理 paused 超过 N 天的 run"，但 M4 / M5 都没真接。M8 上线前补。

- **rollback 删除 last_n 时如果会话只有 < N 条消息**：后端 `Message.delete().limit(n)`
  天然只删可见的；返回 `deleted_messages = 实际删除数`。前端 UI 不预校验，让用户随便选
  N 然后看 SnackBar 提示。

- **streaming 中 pause/cancel 的竞态**：用户在同一秒内点暂停 + 取消，pause 先到，
  状态切到 paused。取消时按当前实现 cancel() 检测 `run.isRunning`，paused 不算
  running → cancel no-op。如果用户想"先暂停再取消"必须先 resume 再 cancel。
  UX 上接受这个简化。

- **archived_branch 会话上长按消息：手势不响应是 by design**，但用户可能困惑
  "为什么这个会话不能复制？"。M5.6 polish 时加 banner 提示
  "这是只读历史分支，复制 / 编辑请在主线"。

- **MarkdownBody selectable=false 副作用**：用户不能拖选 assistant 回答的局部
  文字。M5.4 唯一的复制路径是"长按 → 复制全文"。需要局部选择的场景（如复制
  一段代码块）目前只能复制全文然后手动截取。M5.6 视用户反馈决定。
