## 1. 问题诊断

读 m7-post-m75-baseline/results.json 175 题逐项核对，把 teleqna_transformed
fact_coverage 极低 (0.044) 与 forbidden 误命中归结为 **三类**：

### 1.1 forbidden = MCQ 干扰选项（最致命）

transformer LLM 把原 MCQ 的错误选项直接抄成 `forbidden` 列表 —— 但这些"干扰
选项"在自由文本回答中常常是**正确且必要**的术语：

| item | golden forbidden | 实际语义 |
|---|---|---|
| def-040 | `EAS address information`, `DNN and S-NSSAI`, `AF Identifier` | **正是问的内容本身**（"EAS Mapping 记录里有什么字段"）|
| def-009 | `Transport of NAS messages` | NG-C **本来就**承载 NAS 消息 |
| def-038 | `5G-GUTI`, `GUAMI` | 答 5G-TMSI 时正常会顺带提到这些上下文标识 |
| proc-030 | `Signal quality`, `User preferences` | PLMN 选择的正常考量因素 |
| proc-036 | `5G access stratum time distribution indication` ... | 提问的消息内容里**本就有**这些 IE |

→ teleqna 非 negative 题中 78/116 题有 forbidden（212 条），全部是这类 noise。

### 1.2 expected_facts = 长描述句而非可 substring 短语

transformer 把 TeleQnA 的英文解释抄成"完整意思句"——但 runner 的
`_fact_coverage` 用的是 `f.lower() in answer.lower()` 严格子串匹配，长句
几乎不可能匹配上（连字符/标点/语序任一差异就 miss）：

| 反例 fact | answer 实际写法 | 子串匹配 |
|---|---|---|
| `H-SMF provides VPLMN Specific Offloading Information` | `The H‑SMF provides V-PLMN Specific Offloading Information` | ❌（连字符 H‑SMF vs H-SMF）|
| `gNB-DU includes low layer configuration in INITIAL UL RRC MESSAGE TRANSFER.` | `the gNB-DU includes the low layer configuration ... INITIAL UL RRC MESSAGE TRANSFER` | ❌（前缀 "the" + 句号）|
| `From NF service consumer` | `between an NF Service Consumer and an NF Service Producer` | ❌（语序）|

→ 494 条 fact 里 327 (66%) 是这类长句。

### 1.3 expected_specs 偶错（不在本期范围）

抽样 spec_recall=0 的 69 条 teleqna 题，发现部分确属 golden expected_specs
标错：

- `def-049` SCEF 标 `23.401`，但 SCEF 真正定义在 `23.682`
- `def-029` HSS 标 `23.401 §4.3`，更规范的是 `23.002`

但每条都需要人对 3GPP 原文做归属决定；与 M8 retrieval 调优共用同一份核对，
本期不做。

## 2. 修复策略

### 2.1 forbidden — 全清

teleqna 非 negative 题的 `forbidden` 全部置 `[]`，原列表挪到
`notes` 末尾留审计痕迹。

> 为什么不"挑出真正应该禁的"？因为 transformer 的 forbidden 噪声率 100% 来自
> MCQ distractor 这一来源；保留任何一条都得人工逐项判断成本高。若 M8 真要补
> hallucination 检测，应该走 LLM judge（如已有的 negative_judge）按"contradicts
> retrieved chunk" 这类语义判定，而不是回到子串。

### 2.2 expected_facts — LLM rewrite 为原子短语

用 deepseek-v4-pro（M7.7 刚切的 judge LLM）按 system prompt：

```
output 3-6 atomic key phrases (2-5 words each) that any correct free-text
answer would naturally contain verbatim. drop metadata phrases like "Content
defined in 3GPP 29.502". keep canonical 3GPP capitalization. strict JSON.
```

跑了两轮：
- 第一轮 max_tokens=400 → 大量空 content（v4-pro 是 reasoning 模型，
  reasoning_content 把 budget 占满了），仅 9/116 成功
- 第二轮 max_tokens=4000 → 104/116 成功，仅 4 题 LLM 返回空（保留原 fact
  不动），2 题 LLM 把已经能匹配的短语改坏

2 条手动修正（proc-007 / proc-030）—— 把 LLM 输出与原始 OK 短语合并；其余 102
条自动重写质量都好（spot-check def-002 / def-003 / def-007）。

原 `expected_facts` 全部归档到每题的 `_original_expected_facts` 字段（M8 若
要回滚或对照可用）。

## 6. 给 M8 的 tech-debt 队列

按 ROI 排：

1. **expected_specs 抽样核对** —— 修过 def-049 / def-029 之类的归属错误后，
   teleqna spec_recall 还有 ~0.1-0.2 pp 提升空间（不止 retrieval 问题）。需要人
   + 真协议对照，半人工。
2. **fact_coverage 改 LLM semantic judge** —— substring 仍是死板的；2-5 词原子
   短语已经把它能解决的部分解决了，但 paraphrase 类 fact（"reduces UE power
   consumption" → "improves energy efficiency"）substring 仍 miss。引入
   semantic similarity (cross-encoder 或 LLM judge) 可以再上一个台阶。
3. **forbidden 改 LLM judge** —— 与 fact 类似；当下 forbidden 仅对 hand_crafted
   非 negative 6 题有意义（占总不到 4%），ROI 小。
4. **chunker merge metadata 升级** —— [M7.6 handoff §5.3](./2026-05-23-m7.6-metric-and-golden-fix.md#5.3)
   提过的事；与本期无关但顺带提醒。

## 7. 自主决策记录

- 用 `deepseek-v4-pro` 而不是 `mimo-v2.5-pro` 做 fact rewrite：M7.7 刚切的 judge
  模型同源，且更便宜
- LLM 第一轮失败后没 fallback flash 而是直接 bump max_tokens 4000 重试：v4-pro
  reasoning 占 token 是已知特性，给够空间是直接修法
- 没动 hand_crafted：本次问题信号 100% 来自 teleqna，扩大 scope 是过度修
- 没改 substring 算法本身：M8 应该走 semantic 判定，不是再 patch 子串规则
- `_original_expected_facts` 字段加在每条 item 上保留回溯路径；不嫌占空间
