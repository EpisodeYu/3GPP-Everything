## 3. 关键设计决策

### 3.1 Runner 环境：GH-hosted + fail-soft skip

**问题**：daily/weekly eval 要拿 live LiteLLM/Voyage/Langfuse keys 跑 + 通过 HTTP 打
backend。M8 上线 + HTTPS 之前，backend 只跑在用户本机 localhost；GH-hosted runner
拿不到。

**选项**：

| 选项 | 优点 | 缺点 |
|---|---|---|
| A. GH-hosted + 缺 secrets 自动 skip live（**选中**） | 零外部基础设施；workflow 现在就能 cron 跑 smoke；M8 补 secrets 即 live | 现在不能跑两次完整 live eval（验收条 3 暂挂） |
| B. Self-hosted runner | 立刻能跑 live | 需要服务器 register runner + 给 GH 长 token；运维成本，M8 才会有公网域名 |
| C. 宿主 cron + python | 立刻全链路 | 偏离 `07-cicd.md §2` 已定的 GH Actions 路径，issue 创建要走 gh CLI |

**决策**（plan 阶段人 approve）：选 A。fail-soft 实现：

```yaml
- name: Resolve live-eval mode
  run: |
    if [ -n "${EVAL_BACKEND_BASE_URL}" ] && [ -n "${EVAL_BACKEND_TOKEN}" ]; then
      echo "RUN_LIVE_EVAL=1" >> "$GITHUB_ENV"
    else
      echo "::warning::EVAL_BACKEND_* 未配置 → 仅跑 smoke"
    fi
```

`test_golden_v1_daily` 与 `_full` 已经在 `@pytest.mark.skipif(not _RUN_LIVE)` 守门，
缺 env 自然 skip，smoke 测试仍跑通验证 SSE 契约。

### 3.2 Issue 创建：失败路径 + mock 路径并存

**条件**：

```yaml
if: failure() || (github.event_name == 'workflow_dispatch' && inputs.mock_issue == true)
```

两种触发：

1. **真实失败**：测试 fail / 步骤报错 → `failure()` 真 → 创建标 `nightly-fail` / `weekly-fail` 的 issue
2. **mock 验证**：手动 `gh workflow run eval-daily.yml -f mock_issue=true` → 即便测试
   通过也走 issue 创建路径，开 `[MOCK]` 标题 + label `mock` 的 issue，可手动 close

这样既覆盖了 `06-...md §0 M7.6` "mock 验证：手动塞失败结果触发 issue 创建" 的验收条，
又不需要造一个假失败结果污染历史。

> issue body 包含 `runUrl` 直链到 workflow run，方便排查；阈值定义引到
> `06-...md §7`。

### 3.3 EVAL_REPORT_DIR 环境变量：CI/本地通吃

**问题**：`test_golden_v1_daily` 原实现不写报告（只 assert）。CI artifact 要的
`results.json + report.md` 没人生成。

**选项**：

- A. workflow 额外起一个 step 调 `eval.runner.write_report` — 但 `run_eval` 结果在
  pytest 内存里，跨进程要 dump
- B. 在 test 里 inline 调 `write_report`（**选中**）

实现：

```python
def _maybe_write_report(results, *, default: str) -> None:
    outdir = Path(os.getenv("EVAL_REPORT_DIR") or default)
    try:
        write_report(results, outdir)
    except Exception as e:
        print(f"[eval] write_report failed (outdir={outdir}): {e}")
```

放在 assertion 之前调，保证阈值未达时报告仍能落盘。本地不带 `EVAL_REPORT_DIR` 时
默认路径 `eval-results/m7-daily-latest` / `eval-results/m7-weekly-latest`，覆盖最近
一次跑的结果（人翻历史看 git 日志 + Langfuse trace；本地不需多份）。

## 6. 自主决策记录（CLAUDE.md §4.3）

- 选择 `actions/upload-artifact@v4`（最新 GA）；
- artifact 命名 `eval-{daily,weekly}-${run_id}`；retention 30/90 天（daily 短一些避免堆积）
- `concurrency: cancel-in-progress: false`：评测是带状态副作用的（Langfuse trace 写入、
  spec usage 计费），重叠跑不致命但浪费，宁可不取消已在跑的
- 用 `inputs.mock_issue == 'true'` 字符串比较（GH Actions 把 boolean input 序列化成
  string；如果用 `=== true` 永远 false）
- 不把 `EVAL_REPORT_DIR` 默认路径加 `${ts}`：本地翻看历史靠 git log + Langfuse；CI 用
  `${{ github.run_id }}` 保证唯一性

| 4 | `EVAL_BACKEND_TOKEN` 过期 / 被吊销 | 长期 token 默认 7 天（refresh） | M8 时给 daily eval 创一个 long-lived service token，单独的 user 角色 `eval-bot` |
