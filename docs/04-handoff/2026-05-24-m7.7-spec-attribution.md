## 2. 修复策略

`eval/scripts/golden_repair_spec_attribution.py`：

1. 每条 teleqna 题，按 `teleqna_origin_id` 从 `eval/teleqna/data/raw.jsonl`
   取原 TeleQnA 记录（含 question + correct option + explanation）
2. 用 **mimo-v2.5-pro** 跑结构化 JSON 输出，prompt 让 LLM 综合 (a) explanation
   里的 spec 引用（如 "defined in TS 23.303"）+ (b) 3GPP 通用 canonical-owner
   知识（NR RRC → 38.331、F1-AP → 38.473 等 cheat-sheet 显式列入 prompt）
3. 输出 `{spec_ids, confidence: high|medium|low, reasoning}`
4. **自动 apply**：confidence=high + proposed ≠ current + proposed 非空
5. **不自动 apply**：medium / low（写入 audit 报告留人审）

原 `expected_specs` 归档到每条的 `_original_expected_specs` 字段，与 M7.7
facts/forbidden 修复风格一致；notes 末尾加 `M7.7+ 2026-05-24` 脚注 + LLM
reasoning。

### 2.1 prompt 设计两轮迭代

第一轮 prompt "explanation 必须字面引用 TS XX.XXX 才能 high-conf" → 太严，
116 题里 106 个 low-conf empty list（explanation 多数是概念解释而非规范源
引用）。

第二轮把 prompt 改为"用 explanation + 3GPP 知识联合判定"，并显式列出
NR/LTE/SBA 各概念到 canonical TS 的对照表（让 mimo 不需要"猜"）→ 28-29 个
high-conf diff（每次跑会有 ±1 抖动，因为 mimo temperature=0 但 reasoning
仍有随机性），46 个 medium-conf 待人审，0 个 low-conf empty。

## 3. 数据 — 重算（不重跑 agent）

`eval/scripts/recompute_substring_metrics.py` 扩展支持 `spec_recall` 重算
（直接用 results.json 的 `retrieved_specs` 对新 golden 的 expected_specs[*].spec_id
判命中）。

整体（175 题，对照"M7.7 facts/forbidden 修后" vs "M7.7+ 加 spec attribution 修"）：

| 指标 | M7.5 baseline | M7.7 (facts/forbidden) | **M7.7+ (+spec)** | 累计 Δ vs M7.5 |
|---|---:|---:|---:|---:|
| spec_recall | 0.535 | 0.535 | **0.553** | +1.9pp |
| fact_coverage | 0.111 | 0.496 | 0.496 | +38.5pp |
| forbidden hit% | 18.3% | 14.9% | 14.9% | -3.4pp |

按 source × category（关键 teleqna 子集）：

| key | spec_recall M7.7 → M7.7+ | Δ |
|---|---:|---:|
| teleqna::definition | 0.367 → **0.417** | +5.0pp |
| teleqna::multi_section | 0.667 → 0.667 | 0 |
| teleqna::procedure | 0.475 → 0.475 | 0 |
| teleqna::table_lookup | 0.417 → 0.333 | **-8.3pp** |

#### 为什么提升 modest？

- 29 题被改 expected_specs。要让 spec_recall 0→1 需要：**agent retrieval 早
  就找到了真 canonical spec**（只是当时 golden 标错了而被算 miss）。否则
  改完 golden agent 依然不命中。
- 大量 fix 是 PHY procedures 内移动（38.214 ↔ 38.213）或 service-based
  spec 内挪（29.503 ↔ 29.504）—— 这些 spec 在 retrieval 上语义相近，agent
  可能两个都没拿到或两个都拿到。
- teleqna::table_lookup -8.3pp 是真退步：3 条 high-conf fix 里有 1-2 条把
  expected 改到了 agent 当下检索不到的 spec（如 table-002 38.331 → 38.321、
  table-011 36.213 → 36.211、table-012 23.501 → 22.261）。这反映 retrieval
  本身的弱点（M8 优化空间），不是 golden bug。

**golden 现在比之前更接近 ground truth**；spec_recall 没大涨说明 retrieval
有更多自身空间可挖（M8 接力）。

## 7. 自主决策记录

- 用 mimo-v2.5-pro 而不是 deepseek-v4-pro：用户明确"500M mimo-v2.5-pro
  token"，且 mimo 在中文/3GPP 概念上之前表现稳定；deepseek-v4-pro 是 judge 路线
  保留
- prompt 双轮迭代：第一轮太严 → 第二轮加 cheat-sheet。事后看：cheat-sheet 是
  防止 LLM 太"谦虚"（"explanation 没写就不算"）的关键 anchor
- 自动 apply 只到 high-conf：mid 自动 apply 风险大（46 条里有不少是 "23.501
  vs 23.502" 这种系统架构 vs 系统流程的细分，人判断更稳）
- `_original_expected_specs` 归档：与 M7.7 facts repair 用同一风格，回溯/对照成本低
- 没扩 hand_crafted：用户没要求；hand 已是 reference 高质量
