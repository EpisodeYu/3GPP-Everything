## 3. 核心算法逐讲

### 3.1 `tokenize_utils.py` — Voyage tokenizer

```python
DEFAULT_MODEL = "voyage-4-large"
count_tokens(text)                              # 单文本
count_tokens_batch(texts)                       # 总数
tokenize(text)                                  # 返回 tokenizers.Encoding（含 .offsets）
split_by_tokens(text, max_tokens)               # 按 token 强切，字符边界对齐 offsets
truncate_to_tokens(text, max_tokens)            # 截断到 max_tokens
```

**为什么不用 tiktoken**：
- Voyage 文档明确说 Voyage tokenizer 比 tiktoken 多 1.1-1.2×。chunker 切分阈值若按
  tiktoken 算，会比 Voyage 实际略短，导致 embedding 偶发超长被截断。
- voyageai SDK 的 tokenize 是**纯本地操作**，不调网络。首次跑会从 HF 拉
  `voyageai/voyage-4-large` 的 `tokenizer.json` 缓存到 `~/.cache/huggingface`。
  无 API key 也能跑（传 'dummy'）。
- 性能：30k token 单文本 ~1s；1000 短文本 batch ~21ms。chunker 一篇 spec 调用次数
  受控，性能足够。

### 3.2 `garbage_filter.py` — 两级过滤

| 规则 | 触发 | 例子 |
|------|------|------|
| stop-list 精确匹配 | `_normalize_title(section_title)` ∈ {contents, foreword, copyright notification, ...} | "Contents" / "**Copyright Notification**" |
| spec 模板 H2 | title 含 "3rd Generation Partnership Project" 或 "Technical Specification Group" | "**3rd Generation Partnership Project; ... (Release 19)**" |
| clause 像 spec_id | clause 匹配 `^\d{2}\.\d{3}(-\d+)?$` | clause="38.211"（hf-loader 注意点 6.3） |
| 空段 | body strip 后 < 30 字符 | 残留分隔符 |
| 联系信息 | body < 600 字符且含 "Postal address" / "Sophia Antipolis" / "Internet" | spec 元信息 |
| TOC 表 | 行数 > 20 且 `\|` 起首行占比 > 80% | "# Contents" 的 TOC |

返回 `(kept, dropped, reason_map)`，`reason_map` 进 BuildStats 供 audit。

**实测在生产 spec 上的 drop 命中**（38.331 sections=2704，dropped=310）：
- empty-body: 275（Marker 转换的孤儿空段）
- toc-table: 18
- stop-list: 15
- spec-template-title: 1
- contact-info: 1

### 3.3 `atomic_blocks.py` — body → list[AtomicBlock]

单遍扫描 + lookahead，按优先级识别：

1. ASN.1 块（`-- ASN1START` ~ `-- ASN1STOP`，可带 `/example/`）
2. 图片（`![alt](path)`，吸收紧邻 GSMA 描述段 + `Figure X.Y-N:` caption）
3. 表格（连续 `|` 行 + delim 行 + 上一行如是 `Table X.Y-N:` 抽为 caption）
4. action_list（连续 `- N>` + 4-空格缩进续行）
5. formula_block（`$$..$$`）
6. paragraph（默认）

**原子内切片**（在 `_split_oversized_block` 中调用）：

- `split_table_text(text, max_rows_per_chunk)`：按行切，每片复制 caption + header + delim
- `split_asn1_text(text)`：按顶层 `Identifier ::=` 切，每片头尾保留 ASN1START/STOP
- `split_action_list_text(text, max_tokens)`：递归找最浅 `- N>` 切；超大子片向更深递归，
  最深仍超用 `split_by_tokens` 强切

### 3.4 `section_splitter.py` — 贪心 packing + 三级 fallback + 安全网

输入：经 atomic_blocks 拆好的 `list[AtomicBlock]`，目标 `target_tokens=250`，
`max_tokens=400`，`overlap_tokens=50`。

算法：

1. 顺序遍历 blocks，往 pack 累加；满 `target_tokens` 立即 flush
2. block 超 max → flush pack + 调 `_split_oversized_block`
3. paragraph 超 max → 三级 fallback：双换行段落 → 句子 → `split_by_tokens` 强切
4. figure → 不进 pack；单独 yield 一个 `chunk_type="figure"` piece，留给 builder.py 用 figure.py 重构 content
5. overlap：相邻"纯 paragraph" piece 复制末尾 `overlap_tokens`（按 token 反推到最近句子边界）
6. **最终安全网**：figure 允许超（GSMA 描述天然原子）；其他类型超 `max × 1.5` 强切

### 3.5 `merger.py` — 短 sibling 合并

GSMA spec 大量孤儿小节（如 38.211 5.1.1 BPSK / 5.1.2 QPSK / 5.1.3 16QAM）每个只有
几十 token；不合并会产大量"标题占满"的 chunk。

合并规则：
- 同 `parent_clause`（5.1.* 共享 parent="5.1"）下连续的 section
- 任一 section body < 200 tokens
- clause 必须有"."（顶层 "1 Scope" 不参与）
- 合并后总 tokens > max_tokens 时先封口

合并产物：`SectionBlock(clause="5.1", section_title="<merged: 5.1.1 / 5.1.2 / 5.1.3>", body="### 5.1.1 BPSK\n<body>\n\n### 5.1.2 QPSK\n<body>\n...")`。

实测：38.211 sections=277 → merged=37 → kept=186；38.331 sections=2704 → merged=49。

### 3.6 `figure.py` — GSMA 描述抽取 + vision_resolver 接口

```python
extract_figure(block) -> FigureExtract:
    image_alt          # ![alt](...) 中的 alt
    image_path         # ![](...path...) 中的 path
    gsma_caption_text  # GSMA 自带的英文描述段（多段拼接）
    spec_caption       # 'Figure X.Y-N: ...' 标题

build_figure_content(extract, *, spec_id, clause, section_title,
                     surrounding_paragraph, vision_resolver) -> (content, raw_extra)
```

**plan 方案 Y 的实现**：

- `vision_resolver=None`（M1 阶段默认）：content = `[<spec_id> § <clause> <section_title>]`
  头 + `Figure caption` + `Description: <gsma_caption_text or alt>` + `Context: <surrounding>`
- `vision_resolver` 接口（vision.py 写好后传入）：调用 resolver 拿 JSON
  `{figure_kind, visible_labels, visible_acronyms, description, spec_role}`，
  content 的 description 用 vision JSON 的 description 字段，
  其他字段（labels / acronyms）也注入 content；**结构化字段同时存进 raw_extra["vision"]**
  供 Qdrant payload facet
- resolver 抛异常时：兜底到 GSMA 描述，`raw_extra["vision_error"]` 记录原因

### 3.7 `builder.py` — 主入口

```python
build_chunks(bundle, *, params=None, vision_resolver=None) -> (chunks, stats)
```

工作流：
1. `garbage_filter.filter_sections(bundle.sections)` → kept / dropped / reasons
2. `merger.merge_short_siblings(kept)` → 合并孤儿短 section
3. for each section：
   - `atomic_blocks.parse_atomic_blocks(sec.body)` → list[AtomicBlock]
   - 抽 figure 前一个 paragraph 作 surrounding_paragraph（≤ 600 字符）
   - `section_splitter.split_section(blocks, ...)` → list[SplitPiece]
   - 对每 piece：
     - figure → `figure.extract_figure` + `figure.build_figure_content`
     - 其他 → 头部注入 `[<spec_id> § <clause> <section_title>]\n\n`
   - 生成 `Chunk(chunk_id=uuid5(spec_id|clause|sha256(content)[:16]))`

## 6. 重要注意点 / 坑

### 6.1 Voyage tokenizer 首次跑会下载 `~3MB` 文件

`voyageai.Client.tokenize` 首次访问会从 HF 拉 `voyageai/voyage-4-large/tokenizer.json`，
~3MB，缓存到 `~/.cache/huggingface/hub/`。匿名访问够用；后续命令用本地缓存。

### 6.2 chunk_id 与 dataset_revision 解耦

`chunk_id = uuid5(spec_id|clause|sha256(content)[:16])`，跨 dataset_revision 同 content
→ 同 ID。Qdrant upsert 不会因 revision 变化产重复 point。

但**`source_version` 字段仍存当时的 revision**，便于审计 / 回滚。

### 6.3 `parent_section_id` 需要 section_title 入种子

GSMA marker 解析后部分 Annex / 字母后缀 clause（"5.15.11.5a"）会被 markdown_parser
归入 clause=""；merger 把多个孤儿 sibling 合并到同一 parent clause。这两类情况都
需要 section_title 区分，否则 `parent_section_id` 冲突 → small2big 召回时把不相干
的章节拼成一段。

### 6.4 figure chunk 大小由数据决定

GSMA 自带描述长度差异巨大：23.501 架构图描述可达 2000+ tokens；logo 类只 30 tokens。
chunker 不切 figure（天然原子），content 可能比 max_tokens 大很多。但 Voyage 4-large
支持 32k context，2000 tokens 远低于上限。

### 6.5 action_list 不一定从 `- 1>` 起首

markdown_parser 按 heading 切 section，可能把 RRC procedure 切到中段，导致 section
body 从 `- 4>` 起。`split_action_list_text` 必须找本块**最浅 N**切，不能假设 1。

### 6.6 安全网兜底强切

table / action_list / paragraph 的切片估算可能有边界 case 漏网（特别是宽行表 + 单
`- 4>` 块）。`section_splitter._enforce_size_safety_net` 对超 max × 1.5 的非 figure
chunk 调 `split_by_tokens` 强切。raw_extra 标记 `force_split_overflow=True` 供监控。

### 6.7 短 sibling 合并阈值与 chunk_type 关系

`short_section_threshold=200` 是按 body 原始 token 算（不含头部注入）。合并后的
section 可能仍 > 200，但 splitter 会按 max_tokens 切——这是预期行为。

### 6.8 Voyage tokenizer 与 LLM tokenizer 不通用

chunker 用 Voyage tokenizer 做切分。后续 reranker（Voyage rerank-2.5）也是 Voyage
体系，token 数一致。但 LLM（mimo / Qwen）用各自 tokenizer，prompt 长度估算
不能复用 Voyage count_tokens。

### 6.9 单测里 body 必须 ≥ 30 字符

否则被 garbage_filter 的 `empty-body` 启发式吃掉。test_builder_smoke 已踩过坑。

### 6.10 figure 的 surrounding_paragraph 取 figure block 之前的 paragraph

builder.py 在 split_section 之前先扫 atomic blocks 找 figure 的"前 1 个 paragraph
block"作为 surrounding_paragraph（最多 600 字符）。**section 之外的上下文（前一个
section 的内容）目前不传**，因为 chunker 当前以 section 为最小处理单位。如未来需要
跨 section 的图片上下文，需在 builder 内重组。

## 10. 给后人的笔记

- **真实数据驱动设计**：原 plan 写"500-800 tokens / 120 overlap"是没看真实 spec 拍
  脑袋的；38.211 里随便一个 section 就是 22k tokens。**任何切分策略上线前都必须
  在 38.331（最复杂）+ 38.211（公式表多）+ 23.501（图密集）三类样本上跑过**。
- **Voyage tokenizer 比 tiktoken 略大**：差异只有 10-20%，但 chunker 边界值卡得太
  紧时这点差异就会让 embedding 时偶发超长。**用最终 embedding 服务的 tokenizer
  做 chunker 切分**是经验法则。
- **GSMA raw.md 自带 image description 是个意外发现**：节省了 vision 的 60-70%
  调用成本（如果完全复用），但描述质量参差。最初 plan 选方案 Y（GSMA 当 caption
  注入 mimo），但 v2 benchmark 实测发现注入反而传染 GSMA 错误（如 architecture-rp
  把 N59 错传成 N58），最终改为方案 E（mimo 不接受 GSMA hint，纯看图）。
  教训：**任何"借力"的设计都要先测正确率再上线**。
- **action_list 可以从任何 N 起首**：不要假设 procedure 一定从 `- 1>` 开始。
  markdown_parser 按 heading 切，会把 procedure 切到中段。
- **chunk_id 用 content hash 替代 offset**：是 small2big + 跨 revision 幂等的关键。
  传统 (spec, clause, offset) 在重新 ingest 时会因 offset 漂移产生幽灵 chunk。
- **safety net 是必须的**：所有"按行 / 按 level / 按行 token 估算"的切分都有
  edge case；最后一道 split_by_tokens 强切兜底，是工程上的"fail-safe"。
- **小步走 + 跑真实数据**：本次最大的收获不是某个算法，而是 plan → implement →
  集成测真实数据 → 暴露 3 个 bug → 修 → 再跑一遍的工作循环。**单测都过不代表
  对真实数据正确**，集成测是真理。
