## 2. 校验规则一览

| 规则 | 级别 | 触发条件 |
|---|---|---|
| `yaml_parse_error` | error | PyYAML 无法解析，带 1-indexed 行号 |
| `root_not_mapping` / `missing_items` / `items_not_list` | error | 顶层结构错误 |
| `missing_field` | error | item 缺 `id / category / language / question / expected_specs / expected_facts / forbidden / source` 之一 |
| `empty_field` | error | 必填 scalar 字段是空字符串 |
| `field_not_list` | error | `expected_specs / expected_facts / forbidden` 不是 list |
| `duplicate_id` | error | id 跨 items 重复，message 标首次出现的 items[idx] |
| `invalid_category` | error | category ∉ `{definition, procedure, multi_section, table_lookup, formula, tool, negative}` |
| `invalid_language` | error | language ∉ `{en, zh}` |
| `unknown_source` | **warning** | source ∉ `{teleqna_transformed, hand_crafted}`，但不阻塞（兼容历史别名） |
| `spec_not_mapping` / `missing_spec_id` / `sections_not_list` / `section_not_scalar` | error | `expected_specs[*]` 结构错 |
| `spec_id_format` | **warning** | spec_id 不像 `NN.NNN`（如 `"38331"`） |
| `negative_has_specs` | **warning** | negative 题的 expected_specs 非空；模板要求空数组，但既有 teleqna 转化数据保留了"用来解释为何找不到"的 spec 引用，warning 不阻塞 |
| `negative_must_say_not_found` | error | negative 题缺 `must_say_not_found: true` |
| `non_negative_must_not_found` | **warning** | 非 negative 题写 `must_say_not_found: true` |
| `missing_version` | **warning** | 顶层缺 `version` |
| `file_not_readable` | error | `--file` 路径不存在或不可读 |

退出码：`0` = 0 error；`1` = ≥1 error，或 `--strict-warnings` 下 ≥1 warning。

## 3. 设计决策

### 3.1 行号定位

PyYAML 的 `safe_load` 丢行号信息；用 `yaml.compose(text)` 拿 Node tree 一次扫描 →
建 `{("items", idx): line, ("items", idx, key): line}` map。compose 失败回 0 索引，
报错时回落到 None（CLI 显示 `L?`）。1-indexed 给人类（PyYAML 是 0-indexed）。

### 3.2 negative_has_specs 改 warning（而不是 error）

人审 backtrack：`eval/golden/v1.yaml` 现有 3 条 teleqna 转化的 negative 题
（neg-001/002/003）都带 expected_specs；这些题的语义是"答案明示不存在 / 不被支持，
但解释会引用 spec 来说明语境"。把这降级 warning：
- 既不破坏 v1.yaml 现有数据（避免连带返工）
- 又给手写题作者一个明确提示（`hand_crafted` 来源时建议保持 [] 空）
- M7.1 runner 实装 `must_say_not_found_passed` 判定时，可单独看是否 grounding 在某 spec

### 3.3 unknown_source 改 warning

`SOURCE_ENUM = {teleqna_transformed, hand_crafted}`，但允许扩展（未来若加
`crowd_sourced` / `regression_seed` 不应阻塞 validate）。

### 3.4 没有同步实现 `merge` 子命令

文档 §12 M7.0 同时列了 `validate` 与 `merge` 两个子命令。本次只落 `validate`：
- `merge` 的核心价值是"防 id 冲突"，validate 已经在单文件内做了 id 唯一性校验
- 手写题目前还没攒到 ≥ 20 题，没有真实"合并场景"驱动设计取舍（是否保留 source 字段、
  metadata 是否合并、conflict 时是否 error 还是后者覆盖等）
- 等手写题就位再补 merge，避免无样本调试

## 4. 自测结果

```
$ cd eval && uv run pytest tests/unit/test_validators_golden.py -q
22 passed in 1.01s

$ cd eval && uv run pytest tests/unit/ -q
173 passed in 4.42s

$ cd eval && uv run ruff check . && uv run black --check .
All checks passed!

$ uv run --project eval python -m eval.cli golden validate -f eval/golden/_template.yaml
golden validate: eval/golden/_template.yaml — 4 items
  OK (0 warning)

$ uv run --project eval python -m eval.cli golden validate -f eval/golden/v1.yaml
golden validate: eval/golden/v1.yaml — 119 items
  OK (3 warning)
  WARN  L1212 items[64].expected_specs [neg-001]  negative_has_specs: ...
  WARN  L1233 items[65].expected_specs [neg-002]  negative_has_specs: ...
  WARN  L1253 items[66].expected_specs [neg-003]  negative_has_specs: ...
```

3 个 warning 全部来自 v1.yaml 既有 teleqna 转化数据（neg-001/002/003），符合 §3.2
预期；运行 `--strict-warnings` 会标红，普通 CI 不阻塞。
