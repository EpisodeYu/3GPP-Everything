## 3. 关键设计决策

### 3.1 [hotfix] LiteLLMClient.embed body 双协议兼容

**症状**：M7.5 启动时盘点 retrieval 状态，直接调 backend `DenseRetriever.retrieve()` →
`RetrievalError`：

```
qdrant query_points failed: 400 Bad Request
Wrong input: Vector dimension error: expected dim: 1024, got 2048
```

**根因链**（4 步）：

1. `LiteLLMClient.embed(inputs, dimensions=1024)` 把 OpenAI 标准的 `dimensions=1024`
   塞进 POST body
2. LiteLLM proxy 透传 voyage 时**只认 voyage 自家的 `output_dimension`**，
   `dimensions` 字段被忽略
3. voyage 返回模型默认 2048 维向量
4. backend 拿 2048 维向量去查 d1024 collection → Qdrant 400 →
   `DenseRetriever.retrieve()` 抛 `RetrievalError` → `retrieve_node` 内部
   `except RetrievalError: d = []` swallow → dense 候选始终为空

**影响**：生产 dense retrieval **从 M4.0 起一直 fallback 到 BM25 sparse-only**。
2026-05-20 daily eval `context_recall_section=0.775` 实际是 BM25 only 的成绩；之前
所有 daily / D13 阈值口径都建立在 broken pipeline 上。

**curl 实证**：

```bash
# 错的请求：dimensions=1024 → 返回 2048
curl -X POST .../v1/embeddings -d '{"model":"voyage-4-large","input":["x"],"dimensions":1024}'
# → vec len: 2048

# 对的请求：output_dimension=1024 → 返回 1024
curl -X POST .../v1/embeddings -d '{"model":"voyage-4-large","input":["x"],"output_dimension":1024}'
# → vec len: 1024
```

**修法**（1 行 + 双协议 superset）：

```python
target_dim = dimensions if dimensions is not None else self._settings.EMBEDDING_DIMENSIONS
if target_dim is not None:
    body["dimensions"] = int(target_dim)
    body["output_dimension"] = int(target_dim)   # M7.5 hotfix
```

**为什么并存而不是只切 `output_dimension`**：如果未来切到 OpenAI / Azure
embeddings（不认 voyage 字段），并存对任一上游 schema 都是合法 superset
（未识别字段会被忽略）。

**验证**：

- 2 unit test（`test_embed_passes_dimensions` 加 `output_dimension` 断言 +
  新 `test_embed_uses_settings_default_dimension`）
- `docker cp` 推到 `tgpp-api` container + `docker restart` → 直调
  `DenseRetriever.retrieve('What is AMF?', top_k=5)` 实跳中 5 hits

**上报**：CLAUDE.md §5 没明确"发现生产 bug"条目，但影响 D13 阈值可信度，
M7.5 启动时 AskQuestion 上报 + 人 approve（`fix-then-m75`），先修再做 C.2/C.3/C.4。

### 3.2 [C.2] retrieval 校准选型：wide candidate pool + 保持 rerank_top_k=5

**ablation 结果**（hand_crafted 56 题，40 题非 negative 进入召回指标）：

| config | params | section@5 | spec@5 | MRR | p50 |
|---|---|---:|---:|---:|---:|
| **wide_dense_rerank5** ← 新默认 | d50/s50/rrf60/top80/rerank5 | **80.0%** | **92.5%** | 0.711 | 605 ms |
| baseline_rerank5（旧默认） | d30/s30/rrf60/top50/rerank5 | 77.5% | 90.0% | 0.706 | 610 ms |
| wide_dense_rerank10 | d50/s50/rrf60/top80/rerank10 | 80.0% | 92.5% | 0.713 | 568 ms |
| baseline_no_rerank | d30/s30/rrf60/top50/no-rerank | 75.0% | 85.0% | 0.636 | 484 ms |

**为什么选 `wide_dense_rerank5` 而不是 `wide_dense_rerank10`**：

- D13 主硬指标看 `context_recall_section`（top-5 rerank 后），两个 config 都 80%
- 改 `RERANK_TOP_K=10` 会让下游 `generate_node` prompt context 翻倍（5 chunks → 10）
  → token 成本 + LLM RTT 翻倍；section@10 +2.5pp 增益不进 D13 主指标
- 保持 `RERANK_TOP_K=5` 是 surgical changes 原则（不改下游 generate 接口）

**为什么 RRF k 不调**：narrow(30) / 默认(60) / wide(100) 实测在 rerank 下游
**完全无差异**——rerank 把 RRF 排序完全洗掉，只有"是否在 final_top_n
候选池里"重要。

**改动**（`backend/app/core/config.py`）：

```python
RETRIEVAL_DENSE_TOP_K: int = 50      # 30→50
RETRIEVAL_SPARSE_TOP_K: int = 50     # 30→50
RETRIEVAL_RRF_K: int = 60            # 保持
RETRIEVAL_FINAL_TOP_K: int = 80      # 50→80
RERANK_TOP_K: int = 5                # 保持
```

**守卫单测**：`test_retrieval_defaults_match_m75_calibration` 锁死这些数；
未来改默认值必须先跑 ablation 更新报告，否则破坏 D13 第一档评测对照口径。

### 3.3 [C.3] rerank ablation 实证 + RRF 不可调的发现

**rerank 收益明确**（baseline_no_rerank vs baseline_rerank5）：

| metric | no-rerank | +rerank-2.5 | Δ |
|---|---:|---:|---:|
| section_recall@5 | 75.0% | 77.5% | **+2.5pp** |
| section_recall@10 | 82.5% | 77.5% | -5pp ⚠️ |
| spec_recall@5 | 85.0% | 90.0% | **+5pp** |
| MRR | 0.636 | 0.706 | **+0.07** |

> section_recall@10 反向：rerank 只取 top-K（K=5），10 个位置后面留空 — 实测
> 把 rerank 输出截断到只看 top-5 才公平，这里 baseline_no_rerank 跑 top-50 fused
> 比 rerank top-5 在 @10 自然更宽。这不是 rerank 的问题，是 ablation 口径差异。

**RRF k 无效**（关键发现）：narrow(30) / 默认(60) / wide(100) 在带 rerank 的
config 下完全相同：section@5=77.5% / spec@5=90% / MRR=0.706 / 失败题清单 100% 同。
原因：rerank 重排时只看候选集成员（不看 fused_score），RRF 的排序完全被洗掉。

**结论**：未来调参先调 `dense_top_k` / `sparse_top_k` / `final_top_n`（影响候选池
覆盖），不要花时间在 RRF k 上。

完整对照表 + per-category 拆分 + 失败题清单见
[`../../eval-results/m7-rerank-ablation.md`](../../eval-results/m7-rerank-ablation.md)。

### 3.4 [C.4] test_retrieve_node_p50_latency_under_800ms 改造

**历史问题**（M4.5 / M4.7 / M4.9 / M4.10 多次 flaky）：

- 5 题串行 cold-start，每题首次走真 voyage embed API（外网 RTT ~250ms）+ Qdrant
  query + BM25 query（394k docs 全内存）
- 首题往往因连接池冷启 / TLS 握手 + voyage RTT 偶发抖到 1.5-2s，把 p50 拉高
- 原 `assert p50 <= 800.0` 对 5 题样本 cold-path **不鲁棒**

**选项对照**（plan §4.6 给的两个）：

| 选项 | 做法 | 优 | 劣 |
|---|---|---|---|
| A | 阈值 800 → 1200ms + 注释"CI 物理机噪声允许" | 1 行改 | 仍 flaky；首题真抖到 1.6s 会 fail |
| B | 剔最快/最慢两个 outlier 后取 p50 | 抗噪 | 5 题 - 2 outlier = 3 题取 p50，样本太小 |
| **B 改进**（实选） | warmup 2 题（不计 timings）+ 剩 5 题取 p50 + 阈值 800 → 1500ms | 抗噪 + 样本完整 + 真实路径 | 测试多花 2 题时间 |

**实测对比**（同 hand_crafted 5 题 simple QA，新默认 dense/sparse=50/final=80/rerank=5）：

| 环境 | round1 (cold) | round2 (warm) |
|---|---|---|
| docker container 内部 (warm pool) | p50=1186ms / max=1696ms | p50=587ms |
| host venv + 真外网 voyage RTT | p50=1472ms / max=1967ms | p50≈600-900ms |

设计目标 **800ms** 是 docker network 内部 + warm pool 的目标；test 跑在 host venv
真外网 RTT 下 1500ms 是务实阈值。M8 上线如真稳定到 < 800ms 可再收紧。

**改动**（`backend/tests/integration/agent/test_simple_qa.py`）：

```python
# warmup 2 题（吃 BM25 / voyage / qdrant 连接池 cold-path），不计入 timings
for i, case in enumerate(golden_cases[:2]):
    warmup_state = AgentState(rewritten_queries=[case["question"] + f" __warmup-{i}__"])
    await retrieve_node(warmup_state, deps=deps_real)

# 5 题取中位数 P50
timings_ms = [...]
p50 = statistics.median(timings_ms)
assert p50 <= 1500.0, ...
```

### 3.5 ablation runner 放 backend/scripts/dev/ 而不是 eval/

**为什么**：必须复用 backend `app.retrieval.{dense,sparse,hybrid,rerank}` 的真实
实现 + 生产 `Settings`（dim / collection / bm25_dir / RRF k 等），与生产
`retrieve_node` 完全同构。放在 `eval/` 子项目会重复 30 行 retrieval 调用 + 易漂移。

**注意点**：

- ablation runner 是 dev 工具（标 `scripts/dev/`，ruff per-file ignore E501），
  非生产路径不进 CI 默认跑
- 主入口在 host 上跑：`env LITELLM_BASE_URL='http://localhost:4000/v1' ...
  uv run python -m scripts.dev.retrieval_ablation ...`
- BM25 by_spec 持久化目录在 host `/data/tgpp/bm25/voyage`（backend container
  mount 到 `/data/tgpp`），脚本默认走生产 settings，host 上也可直读
- 19 unit test 只覆盖纯函数 metric + 配置 label + load_golden；retrieval 集成
  路径由 ablation 跑本身验证（不入 unit/integration marker）

1. **ablation runner 放 `backend/scripts/dev/` 而不是 `eval/`**：必须复用 backend
   真 retrieval 层 + 生产 Settings 与 `retrieve_node` 同构（见 §3.5）。
2. **`embed` body 双协议而不是只切 `output_dimension`**：未来如切到 OpenAI / Azure
   embeddings，两个字段并存是合法 superset（见 §3.1）。

   top-5；改 RERANK_TOP_K=10 会让下游 generate prompt context 翻倍（见 §3.2）。
4. **RRF k 不调**：实测 30/60/100 在 rerank 下游被洗掉，无差异（见 §3.3）。
5. **C.4 选 B 改进而不是纯 A 或纯 B**：warmup + 中位数 + 1500ms 阈值组合最稳健
   （见 §3.4）。
6. **`test_retrieval_defaults_match_m75_calibration` 守卫单测**：默认值变动必须
   先跑 ablation 更新报告，加守卫防漂移。
7. **不跑 daily eval 重看新基线**：单次 36 分钟，预期 dense 修复后会显著上升；
   M7.6 CI 第一次 cron 自然跑出新基线，重复跑既耗时又重复（详见 §4.6）。
8. **ablation runner 用 argparse 而不是 typer**：dev 脚本 + 不进 eval CLI；
   保持依赖最小（typer 是 eval 子项目用的，backend 已经有 argparse stdlib）。
9. **container `docker cp` + `docker restart` 推 hotfix**：dev 环境快速验证；
   生产部署走 `docker compose up --build` 重新 build image（不在本 handoff 范围）。
10. **ablation 跑在 host venv（不在 container 里）**：避免 BM25 394k docs 在
    container 内吃额外内存（实测加载占 ~800MB）；host 上跑也能复用同样的
    Qdrant / LiteLLM proxy（都 export 到 host port 4000/6333）。
