## 4. 自主决策记录（CLAUDE.md §4.3）

1. **OpenAPI 覆盖度用 hook 兜底，避免 36 处散点编辑**：M4.0–M4.9 已落地的 36 个路由 decorator 多数没显式写 `summary=`。在 `create_app()` 末尾加 `_autofill_openapi_metadata` 走 `app.routes`，对 `APIRoute.summary == ""` 的路由用 `route.name.replace("_", " ").capitalize()` 兜底，`description == ""` 的回填 docstring 第一段（无 docstring 时 fallback 到 summary）。M4.10 admin 4 个路由仍显式写中文 summary（Swagger UI 中文展示更友好），其它沿用 hook 行为。理由：CLAUDE.md §3 "Surgical Changes" — 不动 36 处既有 decorator；同时 OpenAPI 覆盖度测试以代码而非"约定"把关。
2. **`/health` 保留 `version` 字段**：`tests/unit/test_health.py`（M0 placeholder）断言 `"version" in body`。改 health 实现时同步保留 version，比改测试更不破坏 M0 兼容性，且 `version` 对线上多副本排查有用。
3. **Task runner 用 `asyncio.ensure_future` 而非 `create_task`**：仅为通过 mypy（`Awaitable` vs `Coroutine` 协变），运行时行为完全等价（接到 coroutine 时 `ensure_future` 内部就是 `create_task`）。比加 `# type: ignore` 干净。
4. **`/admin/index/rebuild` 仍受 `admin_crawl` ratelimit 约束**（5/天）：复用现有 §6 bucket，不为 index_rebuild 单独开 bucket。理由：index_rebuild 与 crawl 都是"花资源的 admin 任务"，5/天 在 dev / 单运维场景充足；M8 上线后视用量再细分。
5. **Task subprocess 用 `python -m ingestion.cli pipeline-hf`**：与 §9.1 文档一致；`spec_id=None → 全量`，`force=True → --purge-first`。subprocess stdout/stderr 截尾 8KB 落 `task.log_tail`，避免大日志撑爆 PG。
6. **测试注入 task_runner 桩**：路由侧从 `request.app.state.task_runner` 取，缺省走 `default_task_runner`。集成测灌入闭包 stub 直接更新 SQLite test DB，避免起 subprocess + 真 PG。
7. **`/admin/upload-doc` / `/admin/crawl` 不挂路由**：测试 `assert 404`。这是"路由不存在"而非"500/501"——保持与 §0 "M4 范围内主动推迟" 决策一致；M8 落地这两条路由时直接补 router function 即可（schema、ratelimit bucket `admin_crawl` 已就位）。
8. **`/ready` probe 数据结构选 `dataclass(frozen=True)` + 列表**：而非 `dict[str, ProbeFn]`。理由：dataclass 强制每条 probe 带 `name` + `timeout_s`，单测注入 list 比 dict 直观；超时是每 probe 独立属性而非全局。

## 6. 剩余风险

| 风险 | 估计影响 | 缓解 |
|---|---|---|
| Task runner 是同进程 `asyncio.create_task`：API 重启 → in-flight task 丢失 | M4 上 dev 单点容忍；M8 多用户 / 高频上线扛不住 | 已记录到 `04-backend-api.md §9.2 / Batch D.9（D10）`，M8 上线前换 Redis Streams + 独立 worker；本里程碑只暴露 `index_rebuild`，crawl / upload-doc 一并挪到 M8 |
| `/ready` 默认 probe 真触达上游服务（PG / Qdrant / Redis / LiteLLM），冷启 / 网络抖动可能让 `/ready` 间歇 503 | K8s readiness 频繁 flap | 每 probe 独立超时 3s（asyncio.wait_for）；上游恢复后 `/ready` 立刻回 200，不需重启 API；生产可调 K8s readinessProbe periodSeconds=10 |
| OpenAPI auto-fill 用 fn 名 humanize 出来的 summary 不一定够友好（如 `list_users` → "List users" 还行，但 `pause_run` → "Pause run" 不够具体）| Swagger UI 阅读体验略差 | 不阻塞 M4 验收；M5 前端开发用 OpenAPI generator 生成 dart client 时如发现 summary 模糊，按需补显式 summary（增量补，不要一次扫光） |
| `admin.index_rebuild` audit metadata 把 payload 整体写入 | payload 含敏感参数时审计表会泄露 | 当前 IndexRebuildBody 只有 `spec_id` + `force` 两个非敏感字段，安全；后续若加敏感参数（如外部 API key 透传），需在 admin.py 写 `extra` 时过滤 |
| `proc-005` 检索质量 fail 仍未修 | 单 case 复杂 procedure 召回不到，影响 M7 nightly eval 指标 | 已归档到 Batch C.2（R10/R11/R19 retrieval 校准），M7 启动前由对应里程碑的 agent 数据驱动调参；与 M4.10 无依赖 |
