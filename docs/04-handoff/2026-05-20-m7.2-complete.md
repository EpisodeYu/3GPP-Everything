## 2. Ragas 接入设计要点

### 2.1 单题异常隔离链（M7.2 硬要求）

```
run_eval(..., ragas_scorer=scorer)
  └─ for each item: compute_eval_metrics(item, resp)
       └─ if scorer and resp.answer:
            try:
              scores = scorer.score_item(item, resp)   # ← 内层 try/except
            except Exception:
              log + scores = {}                         # ← 外层兜底
            _apply_ragas_scores(result, scores)
```

`score_item` 内部异常隔离三道闸：

1. **入口**：`empty answer / empty contexts` → 直接返回 `{4个 metric: None}`，不调 `ragas.evaluate`
2. **ragas.evaluate 调用**：`raise_exceptions=False` + 外裹 try/except；任一异常 → log warning + 全 None
3. **score 取值**：`_coerce_score` 把 NaN / None / 非数值都归 None；缺的 metric key 也 None

### 2.2 LLM / embedding 注入（避免 OpenAI fallback）

ragas 默认会读 `OPENAI_API_KEY`；本模块通过 `LangchainLLMWrapper(ChatOpenAI(...))` +
`LangchainEmbeddingsWrapper(OpenAIEmbeddings(...))` 完全注入：
- judge LLM = `glm-5.1` @ `temperature=0`，base_url = LiteLLM proxy
- embeddings = `voyage-4-large`（与生产索引一致），同 base_url

```python
# build_default_ragas_scorer()
judge = ChatOpenAI(model=settings.llm_judge_model, base_url=..., api_key=..., temperature=0)
embed = OpenAIEmbeddings(model=settings.voyage_embedding_model, base_url=..., api_key=...)
```

缺 `LITELLM_API_KEY` → 抛 `RagasError`；缺 ragas / langchain-openai → 同。

### 2.3 contexts / ground_truth 与 runner 一致

| ragas 字段 | 来源 | fallback |
|---|---|---|
| `contexts` | `chunks_rerank → chunks_hit → citations` 的 content/text/snippet | 没原文时退化为 `"{spec_id} §{section_path}"` 占位字符串 |
| `ground_truth` | `expected_facts` 拼接 | 空 → expected_specs 的 spec_id 拼接 → 空 → `"(no ground truth)"` |

## 3. native MCQ runner 设计要点

### 3.1 模型对照口径

- 输入：`eval/teleqna/data/filtered/filtered.jsonl`（17 篇 whitelist 命中的 telecom MCQ；每行一条）
- 对每题 → mimo-v2.5 + glm-5.1 顺序各跑一次 → 解析 `ANSWER: option K` → 与 TeleQnA `answer` 对比
- 单题失败容忍：HTTP retry exhausted / parse 失败 → `predicted=None, is_correct=False`；不挂整批
- 同模型内 RPM-limited + 并发 N（复用 `eval.teleqna.infer._RpmLimiter` + `asyncio.Semaphore`）

### 3.2 模型解析鲁棒性

`parse_mcq_answer(text)` 优先级：
1. 正则 `\boption\s*(\d+)\b` case-insensitive → 取第一个命中
2. 退化：单行裸数字（如 `3` / `(3)` / `3.`）

`parse_correct_option(answer)` 复用同一正则，因 TeleQnA `answer` 形如 `option 3: TS 23.303`。

### 3.3 报告产物

`eval-results/m7-native-mcq/{ts}/`：

- `report.md`：人读对照表（每模型 1 行：total / correct / accuracy / parsed / parse_rate / errors / 用量 tokens / 耗时秒）
- `results.json`：机器读，含每题 predicted / correct / is_correct / error

```
| model      | total | correct | accuracy | parsed | parse_rate | errors |
| mimo-v2.5  |   100 |      62 |    0.620 |     98 |      0.980 |      0 |
| glm-5.1    |   100 |      55 |    0.550 |     97 |      0.970 |      0 |
```

口径：accuracy = correct / total（含 parse 失败题，按答错算），用作 **RAG 端到端 fact_coverage 的下限锚**。

## 5. 自主决策记录（CLAUDE.md §4.3）

1. **Ragas hook 嵌入 `run_eval` 主循环 vs 完事后第二 pass**：M7.1 完成报告 §9 建议"独立 pass 跑"避免单题失败拖死整批；本次选**嵌入主循环 + 单题异常隔离**。理由：
   - 异常隔离硬要求已满足（score_item 内层 try/except + run_eval 外层兜底；3 case 单测覆盖 crash 路径）
   - 第二 pass 需要把 AgentResponse 持久化（runner 当前不存）；改动反而大
   - 单题级注入更利于 M7.3 把 ragas score 同步 push 到 Langfuse（同一循环出 trace_id / score）
   - `ragas_scorer` 是 Optional 参数，不传就是占位 None；mock 测试 / CI nightly 默认不调 ragas
2. **ragas 放进 `[optional-dependencies] ragas` 而非 main deps**：
   - ragas 拖一堆 langchain / datasets / pandas 子依赖（>1GB pip cache），CI / 单测环境不需要
   - eval 主流程（runner / report）零 ragas 强依赖；ragas_eval.py 内 `from ragas import evaluate` 延迟到函数内
   - 真要跑 ragas：`uv sync --project eval --extra ragas`（按需）
3. **backend 的 `ragas>=0.2` 保留不动**：M7.1 baseline 已声明，本次只在 eval 侧加；后续 batch C 技术债 cleanup 时考虑从 backend 删（surgical 原则，不顺手改无关）
4. **Ragas judge / embedding 走 LiteLLM OpenAI 兼容端点**：避免引入 `litellm` 直连或 `langchain-litellm` 第三依赖；`ChatOpenAI(base_url=LITELLM)` 是 ragas 文档标准做法，最少依赖
5. **`_extract_contexts` 没原文时用 spec+section 占位字符串**：避免 ragas 收到空 contexts 时计算无意义；前置 `if not contexts: return all-None` 已先一步处理真正空的情况，占位只对"chunk 有 spec_id 但无 content"的边界生效
6. **native MCQ 单测用 `_FakeChatClient` 复刻 `_LiteLLMChatClient.chat()` 协议而非 monkeypatch**：让端到端测试看起来像生产代码，更易理解；fake client 用 `question:` 行还原 item id 来选预设响应（测试构造 item 时把 question = id 即可）
7. **`MCQ_SYSTEM_PROMPT` 要求严格格式 `ANSWER: option K`**：mimo / glm 都是 reasoning 模型，自由文本可能塞解释段；强格式减少 parse 失败率。`parse_mcq_answer` 仍保留 inline `option N` / 裸数字两种 fallback 兜底
8. **没在 cli.py 加 `eval run --with-ragas` 子命令**：runner 通过 pytest `test_golden_v1_daily/full` 触发（M7.6 CI 链路），CLI 没必要再叠一层；想手动跑 ragas 直接写 4 行 python（`from eval.runner import run_eval; from eval.ragas_eval import build_default_ragas_scorer; asyncio.run(run_eval(..., ragas_scorer=build_default_ragas_scorer()))`）
9. **`_FakeChatClient` 在测试里用 question 文本反查 id**：弱耦合但测试可读；正式 `_LiteLLMChatClient` 不需要这种行为
10. **pyproject.toml `scripts/*.py` per-file-ignores 加 `B008`**：为 native_mcq_runner.py 顶层 `DEFAULT_INPUT = Path(__file__).resolve()...` 类常量赋值与已有惯例对齐

## 7. 剩余风险

- **ragas 0.2 在 reasoning judge 下行为**：glm-5.1 是 reasoning 模型，ragas 评估 prompt 可能让其输出推理段 → ragas 内部解析失败计 NaN。已有 `_coerce_score` + 单题异常隔离兜底；如果 NaN 率 > 30%，可换 `mimo-v2.5`（non-reasoning，与 generate 同源但便宜）或 fallback judge。
- **ragas evaluate 是 sync API（内含 asyncio.run）**：在已有 event loop 内调可能 RuntimeError。`run_eval` 是 async 函数，在 async 上下文里调 sync `evaluate` 实测 ragas 0.2 用 `nest_asyncio` 兼容（依赖见 lock）；如果未来升级 ragas 主版本，回归测要包含一遍 ragas 真调用。
- **native MCQ runner 每模型顺序跑**：双模型 100 题约需 8-15 分钟（每题 2-4s × 2 模型，并发 8）。spike 用 `--limit 20`；想加并发把 `--concurrent 16` 上调即可。
- **filtered.jsonl 与 v1.yaml 没 1-1 映射**：native MCQ 跑的是 filtered.jsonl（含 200+ MCQ），与 v1.yaml 的 119 题"经过转化的 RAG 题"不同口径。两边各自报告，对照"裸 LLM vs RAG"差距时按题型 / 主题做分桶对比，单纯均值会失真。本期不强约束分桶；M7.5 batch C 再看是否要补 cross-section 报告。
