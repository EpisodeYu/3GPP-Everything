### 引用 chip 设计

```
markdown text: "PDU Session 流程见 [23.501 §5.6.1 ¶0]。"

CitationInlineSyntax 正则：\[(\d+\.\d+) §([\d\.]+) ¶(\d+)\]
  ↓ 匹配 → md.Element('citation', attrs={spec, section, rank, raw})
CitationElementBuilder.visitElementAfterWithContext(...) → CitationChip widget
  ↓ tap → showCitationBottomSheet → 拉 GET /chunks/{chunk_id} → 显示 content + "跳到完整章节"
  ↓ longPress → Clipboard.setData(raw) + SnackBar('已复制引用文本')
```

要点：

- 正则要求 spec 是 `\d+\.\d+`、section 是 `[\d\.]+`、rank 是 `\d+`：严格不与 markdown link `[text](url)` 冲突（后者括号后必带 `(`）
- chip 通过 `citations[rank]` 拿到真实 `chunk_id`；citations 缺失时 sheet 显示 "无 chunk_id" 占位但仍可跳 reader（按 spec + section）
- "跳到完整章节" 按钮 push 到 `/reader/{spec}/{section}#chunk-{chunk_id}`，由 router `_parseChunkAnchor` 解析回 `activeChunkId`

### Reader 设计

```
/reader/{spec}              → AppBar(spec) + 左 TocDrawer + 中央 _SpecOverview（spec 元信息 + 章节列表 ListView）
/reader/{spec}/{section}    → AppBar(spec · §section) + 左 TocDrawer + 中央 SectionView（标题 + chunks 卡片）
/reader/{spec}/{section}#chunk-{id}
                            → 同上，但 SectionView 加载完后 ensureVisible 到该 chunk + HighlightOverlay 3s 淡出
```

- 响应式：宽 (>= 840) → 固定 toc + 内容 split；窄 → AppBar menu → Drawer 抽屉化 toc
- AppBar 左侧（窄屏 menu / 宽屏 back→/chat）、标题 breadcrumb
- TocDrawer：spec header（id + release/series）+ 搜索框（自动 trim，空 → 章节树；非空 → 搜索结果列表 ILIKE `q`）
- SectionView：`ListView` 渲染每个 chunk（chunk_type badge + chunk_id + MarkdownBody）；`activeChunkId` 改变时通过 `GlobalKey + Scrollable.ensureVisible` 滚到该 chunk
- HighlightOverlay：`AnimationController(3s)` + `Tween(0.32 → 0)` 把 accent 色透明度从 32% 淡到 0；进入时立刻 forward(from: 0)

## 自主决策记录（CLAUDE.md §4.3）

1. **Reader 路由不入 ShellRoute**：docs §4 示例代码里 reader 在 ShellRoute 内，但 AppShell 已经有 Drawer，reader 内部再嵌一个 Drawer 会出现"双 Drawer"+"双 AppBar"，UX 与视觉都不干净。改为平级 GoRoute，reader 全屏 + 自带 AppBar（含 back 按钮回 /chat）；sidebar 阅读器入口通过 `context.go('/reader/{spec}')` push 进去。如果 M5.5 admin 也走类似全屏模式，把模式定型成"主区 = chat(shell 内) / reader / admin"的三态。
2. **不引 `url_launcher`**：docs §3 列了它，但 M5.3 只做内部跳 reader 路由，没有外链需求；M5.5 admin 跳 Langfuse 时再加，符合"按里程碑加依赖"的纪律。
3. **不引 `fluttertoast`**：docs §3 列了它，长按复制提示用 Flutter 自带的 `ScaffoldMessenger.showSnackBar` 完成（在 Material 主题下视觉一致），不需要额外 dep。
4. **fragment 而非 query 传 chunk anchor**：用 `#chunk-{id}` 而非 `?chunk={id}`。go_router 14.x 的 `state.uri.fragment` 在 path 切换时即使路径相同也会触发 builder 重建（ReaderPage 用 `ValueKey(spec-section-anchor)` 强制 SectionView 重 mount → 重新 trigger ensureVisible）。如果用 query，path 不变时 builder 不重建。
5. **chunk anchor 高亮用 `Tween(0.32 → 0)` 而非"亮起 → 维持 → 淡出"三段**：单段 ease 出动画足够"提示用户视线锚点"且实现简单；如果 UX 反馈 "刚跳过去就开始淡看不清"，改为 `TweenSequence([亮 0.5s 维持 + 2.5s 淡出])`。
6. **TocDrawer 搜索本地 trim + setState 触发 docSearchProvider 重 watch**：避免 200ms debounce 引入 timer 让 widget test 时序复杂；用户每输一个字就调一次后端搜索（ILIKE 单 spec 内最多几千 chunk，响应在 100ms 内），交互体感够。如果 M7 上线后看到 search QPS 偏高再加 debounce。
7. **`_SpecOverview` chunks 不预拉**：spec overview 页只列章节 metadata（来自 docDetailProvider），不拉 section content。用户点章节才触发 sectionDetailProvider，节省一次性带宽。
8. **`activeChunkId` 走 SectionView 内的 `_lastTriggeredId` 防抖**：避免 widget rebuild 时一次 frame 多次 trigger ensureVisible 撕扯滚动条；仅在 `widget.activeChunkId != old.activeChunkId` 时重置防抖标记。
9. **citation chip key 用 `citation_chip_{spec}_{section}_{rank}` 复合 key**：方便 widget test 直接 find；同一个 message 内同一引用最多出现一次（rank 唯一），不会 key 冲突；同一 ChatPage 内多条 assistant message 共享同一 rank 时 key 也会重复——但每条 message 已经有自己的 ValueKey 父级，find.byKey 在 widget tree 范围内仍能命中第一个，对当前测试足够。
10. **doc picker 客户端过滤**：1270 篇全部拉来，输入框输 `lower(spec.specId).contains(q)` 过滤；后端 `GET /docs?release=&series=` 已经支持服务端过滤，但 picker 体验上"用户输什么都立刻刷新"比"每输一个字发一次请求"顺滑。如果 M6+ 文档量到 5000+ 才考虑切服务端搜索。
11. **`CitationChip` 用 `Material + InkWell + StadiumBorder`**：MarkdownBody 的 builder 把 widget 包成 WidgetSpan 嵌入 RichText，普通 Container 没有 InkWell 反馈；显式 Material(...) 才能拿到 ripple。
12. **不在 `StreamingAssistantBubble` 里挂 CitationInlineSyntax**：流式 token 期间 citations 还没到（final event 才有），渲染 chip 也是"无 chunkId" 占位；交互上让用户先看完整答案再看引用更自然。

## 剩余风险 / 已知问题

- **doc picker 大列表无虚拟搜索高亮**：1270 个 ListView.builder 虚拟化够用，但搜索关键字不在结果里高亮（仅过滤）；M5.6 polish 时加 `RichText` 高亮匹配段。
- **`#chunk-{id}` 锚点失效场景**：如果 URL fragment 写错（如 `#chunkid-x` 漏 `-`），`_parseChunkAnchor` 返回 null，section 正常加载但不高亮，无用户提示。M5.4 / M5.6 polish 时考虑加 "无效锚点" toast。
- **section view 拼章节内容是把 N 个 chunk 各自渲染**：而非"合成一段连续 markdown 再渲染"——好处是每个 chunk 边界清晰、有 chunk_id 锚点 + chunk_type badge；坏处是跨 chunk 的 markdown 续接（如表格跨 chunk 拆开）会断。M7 看真实语料反馈再决定是否合并。
- **bottom sheet 拉 chunk 没有 401 兜底重试**：dio interceptor 已经处理 401 → refresh → retry，但 sheet 第一次拉如果 refresh 也失败会显示 "加载失败：DioException"；用户得手动关 sheet 再点 chip 重试。M5.5 / M5.6 polish 时加 retry 按钮。
- **`_DocPickerDialog` 第一次打开慢**：1270 篇要等后端拼 response。MVP 接受；如果 UX 反馈太慢，加 sidebar 启动后台预拉。
