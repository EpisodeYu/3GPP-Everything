## 2. 砍 tool 的理由（5 角度拷问）

### 2.1 模板自爆 — runner 不验证工具被调用

`_template.yaml` 示例 3 原 notes 自己写：
> "runner 不直接断言"工具被调用"，靠 expected_facts 间接。"

→ category 标签对评测**无任何区分作用**；评测只看 `expected_facts` substring 命中。

### 2.2 边界与 4 个 category 全部塌方

agent 的 4 个工具一一对应非 tool category：

| Agent 工具 | 已被哪个 category 覆盖 |
|---|---|
| glossary | **definition**（"AMF 是什么"完美装下）|
| params | **table_lookup**（参数表查询）|
| toc | **multi_section**（列子节）|
| web_search | 本期 GSMA 范围外，不该出现在金标准 |

→ tool 实际是"以上 3 类的别名集"，没自己的语义。

### 2.3 路径覆盖是 unit test 的事，不是金标准的事

如果想验证 `classify → tool_dispatch → generate` 边走得通，正确做法是
`backend/tests/integration/agent/test_tool_dispatch.py`，断言 trace_node_names
包含 `tool_dispatch`。M4.4 已经做过。金标准的职责是 **质量评测**
（fact_coverage / faithfulness），不是 **路径覆盖**。

### 2.4 agent 自己决定走不走 tool 路径，标签会误导写题

同一道 "AMF 是什么"题：
- agent 走 simple 路径 → retrieve glossary chunk → 答对
- agent 走 tool 路径 → glossary 工具命中 → 答对
- **两条路径 `expected_facts` 命中完全一样**

写题人标 `category: tool` 无法约束 agent 的内部决策，反而埋雷。

### 2.5 工作量账

stats 之前显示 `tool: 0/10 GAP`。砍掉：
- v1.yaml 当前 0 题在 tool（stats 已确认），删 category 不影响既有数据
- 写题人少写 10 题，22 题需求降到 12 题；hand_crafted ≥ 20 daily 子集仍达标（multi-9 + formula-13 + negative-16 = 38 题菜单挑 20）
- 模板示例 3 改成第二个 formula 例（TBS 公式 vs CQI/MCS 表，强化"NR vs LTE 不要混"）
