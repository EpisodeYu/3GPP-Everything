## 2. 生成 / 手写（source 维度）

| 指标 | hand_crafted (n=56) | teleqna_transformed (n=119) | gap |
|---|---|---|---|
| spec_recall | **0.875** | 0.420 | +0.455 |
| section_recall | **0.800** | 0.303 | +0.497 |
| fact_coverage | **0.315** | 0.044 | +0.272 |
| faithfulness | 0.764 (n=23) | **0.928** (n=34) | -0.164 |
| answer_relevance | 0.781 | 0.820 | -0.039 |
| p50 latency | 35.2s | 20.5s | +14.7s |
| forbidden hit rate (全部) | 46.4% | 5.0% | +41.4 pp |
| forbidden hit rate (非 neg) | 0/40 = **0.0%** | 13/116 = **11.2%** | -11.2 pp |

**关键解读**：

1. **召回率 hand_crafted 完胜**：hand spec=0.88 / sec=0.80 vs teleqna spec=0.42 / sec=0.30。原因是 hand_crafted 题面就是 3GPP 术语（"AMF handles X"），retriever 命中很自然；teleqna_transformed 是机翻多语义跳跃题面（"What is described as ..."），稀疏检索 BM25 和稠密检索都难以对齐。**这不是检索管线弱，是 input distribution 难。**

2. **faithfulness 反向**（teleqna 0.928 > hand 0.764）：teleqna 由于召回经常错（chunk 跟问题没真正相关），agent 仍按错 chunk 写出"忠实"答案（ragas 评忠实于 context 而非真相）；hand 召回准，但 generation 偶尔在公式 / 表格类编数字（hand-table-007 faith=0.0 是极端案例）。

3. **forbidden hit rate 真相**：hand 全量 46.4% 主要被 negative 子集（16/16 命中）拉爆——negative 题的设计是问 fake 概念，agent 拒答时必然复述这些 fake 词作否认（"BGP 不在 3GPP 范围内"），forbidden 把这当违规。**非 negative 子集真实值 hand=0%, teleqna=11.2%。** 当前 forbidden substring 检查机制对 negative 类已知误判，与 06-md §3.4 早有结论。

### INVALID 失败 case (待修)

1. **neg-001** (teleqna, en)
   - 问题：UE 收到 DCCH/CCCH 上 *缺失必需 IE* 的 RRC 消息时的"规定行为"
   - judge reason: agent 把伪命题当真，编造了 "explicitly unspecified for NR" 的规范引用（36.331 §5.7.4 / 38.331 §10.4）
   - 模式：fake premise → 编造 NR/LTE 差异化解释

2. **hand-neg-013** (hand, zh)
   - 问题："X4 handover" 跨 PLMN 触发流程
   - judge reason: agent 把伪概念 X4 偷换为真实存在的 Xn handover，详细描述 Xn 流程，未指出 X4 不存在
   - 模式：fake concept → 名字相近真概念偷换

→ **共同模式**：当伪命题"近似"一个真规范点时，agent 倾向于"修正"而非拒答。M8 prompt 可考虑加强"严格拒答 fake premise，不要语义改写"的指令。

## 6. 失败聚类与可操作 finding

### 6.1 召回二极分布（无 partial credit）

在 156 个非 negative 题中：
- spec_recall = 1.0：**83 (53.2%)**
- spec_recall = 0.0：**73 (46.8%)**
- partial (0 < r < 1): **0 (0.0%)**

→ retrieval 是"要么找对要么完全找错"，没有"找对一半"的灰色态。说明 expected_specs 大多是单 spec，rerank_top_k=5 命中即满分。

### 6.2 召回到位但生成不忠实（spec=1 ∧ sec=1 ∧ faith ≤ 0.5）

**这是 M7.5 后剩余最重要的失败类型**：retrieval 没问题了，generation 还在编。仅 3 条：

| id | category | source | faith | 问题摘要 |
|---|---|---|---|---|
| `hand-table-007` | table_lookup | hand | **0.00** | PUSCH MCS=16 64QAM 目标码率与频谱效率 |
| `hand-proc-002` | procedure | hand | 0.44 | PUCCH format 0 跳频 m_cs 计算 |
| `hand-formula-005` | formula | hand | 0.50 | 多 TCI-State PUSCH 传输功率计算 |

**模式**：全部是公式 / 表格 / 多分支条件类题，agent 选择了"详细列举所有分支"但偏离了 chunk 实际内容。建议下个里程碑做小规模 cause analysis（手工对照 chunk 与 answer）。

### 6.3 非 negative 触发 forbidden（16 条）

非 negative 类的 16 条 forbidden 命中，hand=0 全部来自 teleqna，主要在 definition / procedure，常常是 agent 引用了 question 里出现过的 forbidden 概念。需要审视 forbidden 词表是否包含了应当被允许的"对照否认"用法。具体清单见 `results.json`。

### 6.4 fact_coverage 极低（0.11）

- hand_crafted 非 neg：0.315
- teleqna_transformed 非 neg：0.044
- 表格类（fact 是具体数值）：0.27 — 命中即满分

teleqna 的 fact_coverage 极低主要因为 expected_facts 是机翻泛化文本片段，与 chunk 实际文本词法不对齐；fact_coverage 用 substring 检查所以漏掉。这是 expected_facts 标注口径问题，不是 generation 真错。建议 M8 重审 teleqna 的 expected_facts 标注。
