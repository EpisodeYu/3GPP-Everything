## 2. 设计要点

### 2.1 为什么单独建 eval 子项目的 Langfuse client

不复用 `backend/app/agent/langfuse_handler.py`，原因有三：

1. **`eval` 子项目独立 venv 不 import backend**（沿用 M3 决策）
2. **backend chat 路径当前未挂 `CallbackHandler`**（`grep build_callback_handler` 在 `app/api/v1/chat.py` 0 命中）—— eval 跑 backend SSE 时不会产生 backend-side trace；evaluator 需要的 trace + IO 必须由 runner 自建
3. **缺 key / SDK / 网络异常的优雅 disable** 需要 eval 自己的单例状态（独立 `_init_failed` 短路）

口径与 backend 一致：模块级单例 + 双检锁；任何 SDK 异常吞掉转 log，runner 主路径不受影响。

### 2.2 trace_id 幂等：基于 (run_label, item_id) seed

```python
trace_id = client.create_trace_id(seed=f"{run_label}:{item_id}")
# 同一 (label, id) 多次跑 → 同一 trace_id
```

好处：

- 同一 eval run 多次跑（如 daily CI 重试）不会产生重复 trace 占空间
- 历史 score 沉淀在同一 trace 上，方便看趋势
- 同一 item 跨 run 看（如 m7-daily-2026-05-20 vs m7-daily-2026-05-21）trace 不同，便于对比

### 2.3 score 字段映射

`_result_to_langfuse_scores(result)` 输出（9 个 NUMERIC）：

| Langfuse score name | EvalResult 字段 | 值映射 |
|---|---|---|
| `context_recall_section` | `result.context_recall_section` | float / None |
| `context_recall_spec` | `result.context_recall_spec` | float / None |
| `fact_coverage` | `result.fact_coverage` | float / None |
| `must_say_not_found_passed` | `result.must_say_not_found_passed` | bool → 0/1 / None |
| `forbidden_violation` | `result.forbidden_violations` | `1.0 if list else 0.0`（永不 None） |
| `ragas_faithfulness` | `result.ragas_faithfulness` | float / None |
| `ragas_answer_relevance` | `result.ragas_answer_relevance` | float / None |
| `ragas_context_recall` | `result.ragas_context_recall` | float / None |
| `ragas_context_precision` | `result.ragas_context_precision` | float / None |

`push_run_score` 内部对 None / NaN 自动 skip（log debug），不写无意义 NUMERIC=0 误导聚合。bool → 0/1 是因为避免 v4 `BOOLEAN` 类型的额外 enum 校验，统一 `NUMERIC` 便于 Cloud UI 配阈值 evaluator。

### 2.4 异常隔离三道闸

```
push_golden_to_langfuse
  ├─ get_client() 失败 → 0
  ├─ _ensure_dataset() 失败（已存在 409 等）→ 吞掉，继续推 item
  └─ 单条 create_dataset_item 失败 → log + 跳过这条

push_run_score
  ├─ client None / trace_id 空 → 0
  ├─ value None / NaN → skip 该 metric
  └─ 单个 create_score 失败 → log + 跳过该 metric

run_eval (langfuse_run_label 启用时)
  ├─ get_client() None → 全 disable，langfuse_trace_id=None，行为同未启用
  ├─ make_eval_trace_id None → 跳过该 item 的 event + score
  └─ create_event 失败 → log + 仍尝试 push_run_score
```

任一层失败都不挂 runner 主路径（fact_coverage 等业务 metric 照常算 + 落盘）。

### 2.5 dataset_item upsert 字段映射

| Langfuse 字段 | GoldenItem 来源 | 备注 |
|---|---|---|
| `id` | `item.id` | 幂等键 |
| `input.question/category/language` | 同名字段 | evaluator 读 input 取 question |
| `expected_output.expected_facts` | `item.expected_facts` | evaluator 算 faithfulness/recall 的 ground truth |
| `expected_output.expected_specs` | `[{spec_id, sections}]` | 列表化（namedtuple 不能 JSON 序列化） |
| `expected_output.forbidden` | `item.forbidden` | 给 forbidden_violation evaluator 用 |
| `expected_output.must_say_not_found` | `item.must_say_not_found` | 给负样本评测用 |
| `metadata.source/teleqna_origin_id/notes` | 同名字段 | Cloud UI 按 source 筛 hand_crafted vs teleqna_transformed |

## 4. 自主决策记录（CLAUDE.md §4.3）

1. **不复用 backend langfuse_handler**：理由见 §2.1。surgical 原则 — eval 子项目不应 import backend。
2. **trace_id 幂等用 (label, item_id) seed**：plan §4.4 没指定 seed 形式；选 `f"{run_label}:{item_id}"` 让历史 score 沉淀同一 trace，CI 多次重试不污染 Cloud 配额。
3. **trace + event 而非 dataset_run**：Langfuse v4 没暴露 `create_dataset_run`；`run_experiment` 要求 task 是 sync 函数，重构 run_eval 改动太大。改用 trace_id + `create_event` 写 IO 是 v4 推荐做法，evaluator 仍能正常读 trace IO。
4. **9 个 metric 全 NUMERIC（bool → 0/1）而非用 BOOLEAN**：v4 BOOLEAN 类型有额外 enum 校验；统一 NUMERIC 便于 Cloud UI 配阈值 evaluator（如 `must_say_not_found_passed > 0.5`）。
5. **`langfuse` 放 main deps 不放 optional**：runner 启用时即需要；SDK 体积可接受（不像 ragas 拖 langchain）；缺 SDK 时 import 异常被吞，等同缺 key。
6. **`run_eval` 加 `langfuse_run_label / langfuse_dataset_name / langfuse_client` 三个可选参数**：caller 显式传 run_label 才启用，避免 mock 测试 / unit 路径意外发起网络调用；`langfuse_client` 仅给测试注入用，生产路径走 `get_client()`。
7. **EvalResult 加 `langfuse_trace_id` 字段**：落 `results.json` 让事后能追到对应 Langfuse trace（不接 Langfuse 时 None，不影响序列化）。
8. **没加 `eval.cli golden push-to-langfuse` 子命令**：plan §4.4 文档没要求 CLI 入口；首次推送由人在 REPL 手触发（见 §6）。M7.6 接 CI 时再考虑是否要 CLI。
9. **Makefile `eval-daily` / `eval-weekly` 暂不传 `langfuse_run_label`**：避免每次 daily 都吃 Cloud 配额；M7.6 接通 CI 时统一启用（用 env var 如 `EVAL_LANGFUSE_RUN_LABEL` 注入）。
10. **`ruff format` vs `black`**：项目 `pyproject.toml` 配的是 `[tool.black]`（line 100）；用 black 不用 ruff format（与 backend/ingestion 同口径）。
11. **运行时 import**：`runner.py` 把 `from eval.langfuse_dataset import ...` 放到 `run_eval` 函数体内部（`if langfuse_run_label is not None:` 分支内）；让 unit 测试在没装 langfuse 的极端环境下仍能 import runner（虽然主依赖已声明，但运行时延迟更鲁棒）。

## 6. 剩余风险

| 风险 | 触发 | 缓解 |
|---|---|---|
| Langfuse Cloud 国内访问抖动 | langfuse.com 偶发慢/超时 | 单 metric 失败被吞，仅 log；不阻塞主 runner |
| backend chat 路径未来接 CallbackHandler 后产生两条 trace（eval client 端一条 + backend 一条） | M5/M8 接 backend trace | 用 `langfuse_session_id` 关联两条；或 eval 改成不发 event 让 backend trace 持有 IO（M7.6 决定）|
| 二次推送 dataset_item 字段被覆盖 | 人改了 v1.yaml 某条后重推 | 这是 upsert 设计意图；如需保留历史可在 Cloud UI 打 dataset version |
| `tgpp-golden-v1` dataset name 与 Cloud 已建的同名 dataset 冲突 | 人先在 UI 建过 | `create_dataset` 抛错被吞掉；后续 `create_dataset_item` 仍写到同名 dataset，行为符合预期 |
| Cloud 端 score upload 量级 | daily 56 题 × 9 score × 30 天 ≈ 15k score/月 | 远低于 Free Tier 50k score 限额；月底看 dashboard 监控 |
