**临时兜底**：indexer pipeline 入口 `_dedupe_chunks` 按 chunk_id 去重保留 first occurrence；Qdrant/BM25/PG 看到的是 dedupe 后 8853 个 unique。**检索质量不受影响**（重复 content embed 一次还是多次，向量相同）。

**后续行动**：开 chunker bug ticket 查 splitter 重复输出，修复后 indexer 入口 dedupe 仍可保留作为 belt-and-braces。

**FIX（2026-05-16）**：分两层根因。

1. **同 section 内重复**（占 ~186 / 189）：splitter packing 后 emit 的相邻 `SplitPiece` 偶发 content 完全相同（多见于短 action_list 短语在嵌套结构中重复、被三级 fallback 合并后等价）。`ingestion/chunker/builder.py::_build_section_chunks` 出口加 `seen_content: set[str]` 去重保留 first occurrence。
2. **跨 section 同 parent_section_id 撞 chunk_id**（剩 3 / 189）：GSMA marker 把 `***field***` 等字段名渲染成裸 `####` 标题（clause=""，多处出现 title=`***duration***` / `***offset***`），uuid5(spec_id|clause|title) → 同 parent_section_id；section 内描述一致 → uuid5(spec_id|clause|sha256(content)) → 同 chunk_id。`ingestion/chunker/builder.py::build_chunks` 出口再加一层全局 chunk_id dedupe。

回归：真 spec 38.331 复跑 dup chunk_id **189 → 0**、intra-section content dup **3 groups → 0**。新增单测：
- `tests/unit/test_builder_smoke.py::test_build_chunks_dedup_identical_content_within_section`
- `tests/unit/test_builder_smoke.py::test_build_chunks_dedup_cross_section_same_parent_id`

### 6.2 [chunker P2] 7% tables 缺 markdown `|-|-|` separator

**症状**：106 / 1570 table chunks 没有任何 separator 行（开头直接是 body row）。

**影响**：前端 `react-markdown + remark-gfm` 会失败渲染这些表格，回退到纯文本展示。

**根因**：chunker 在切分大表格时（>1 chunk），第二片以后没有把 caption + header + separator 重新复制。Docs §4.3 设计要求"超长按行切，每片重复 caption + 表头 + delim 行"，实际未完全落地。

**后续行动**：开 chunker bug ticket 修 `section_splitter.py::split_table`，确保每个切片都带 `| ... |` 头 + `|---| ... |` separator。

**FIX（2026-05-16）**：三层根因都修：

1. **行数估算粗糙 → safety net 兜底**：`_estimate_max_rows_for_table` 用 mean row tokens 估算，遇含极长 cell 的表（如 38.331 field description 表）p95 显著高于均值 → `split_table_text` 仍产出超 `max_tokens × 1.5` 的片 → 进 `_enforce_size_safety_net` 走通用 `split_by_tokens` → 头部丢失。改用 p95 row tokens + 显式 header overhead 重算行预算（`section_splitter.py::_estimate_max_rows_for_table`）。
2. **safety net 对 table 类型不带头**：即使 (1) 收敛到极少，仍可能漏；为 table 类型新增专用兜底 `_force_split_table`：拆出 caption / header_block / data_rows，每个子片重新 `assemble(header_prefix + rows)`，单行超预算时按 token 强切该行内容但仍带头部。
3. **`_TABLE_DELIM_RE` 误识空 cell 为 delim**：原正则 `[:\- ]+` 允许全空格 → GSMA 偶发把 field-description 表头渲染为"空 cell 行 + 真 delim 行"双行结构时，空 cell 行被认成 delim，`split_table_text` 把后续 piece 的真 delim 丢掉。改为 `^\s*\|?\s*:?-+:?\s*(\|\s*:?-+:?\s*)*\|?\s*$` 强制每个 cell 至少含一个 `-`（同步 `atomic_blocks.py` + `section_splitter.py` 两个 copy）。

回归：真 spec 38.331 复跑 table no_separator **129 → 0**（其中 force_split_overflow 子集 23 → 0）。新增单测：
- `tests/unit/test_atomic_blocks.py::test_table_delim_regex_rejects_empty_cell_row`
- `tests/unit/test_atomic_blocks.py::test_table_empty_header_row_split_keeps_real_delim`
- `tests/unit/test_section_splitter.py::test_oversize_table_force_split_preserves_header_separator`
- `tests/unit/test_section_splitter.py::test_oversize_table_with_single_huge_row_each_piece_keeps_header`

### 6.3 [vision P3] mimo 偶发 `figure_kind=undescribable` 误判

**症状**：64 张图中 2 张首跑 mimo 返回 description="No figure was provided"。手工删 Redis 缓存后 retry，100% 成功。

**根因**：mimo-v2.5 模型的 false-negative，与图片本身无关（可下载、可显示）。

### 6.5 [chunker P3] section_title 偶发超长（1 / 9042）

**症状**：clause B.2 某 chunk 的 section_title 是整段 1639 字符的 markdown 表头。

**根因**：chunker 把无显式 section heading 的 markdown 表格当作"独立 section"，取表头第一行做 section_title 但没有截断。

**临时缓解**：本次把 `pg_writer.chunks_meta.section_title` 改 `Text`（unbounded）已能写入，但前端 reader 看到这个 section_title 会显得奇怪。

**后续行动**：chunker `_make_section_block` 处可考虑给 section_title 加 200 字符截断 + ellipsis。

**FIX（2026-05-16）**：根因 + 兜底两步走。

1. **根因（pseudo-heading 过滤）**：GSMA marker 把表格首行（`| ... | ... |` 或 `|---|---|`）偶发渲染成 `####` 标题。`ingestion/hf_loader/markdown_parser.py` 新增 `_is_pseudo_heading()`：title 以 `|` 起首 或 内含 `|:?-{2,}` 模式 → 跳过该 heading，正文回归到上一级 section body（让表格被 `atomic_blocks` 正常识别为 table block）。
2. **兜底（长度安全网）**：仍残留漏网的 pseudo-heading（如未来出现其他 marker 异常）时，`_truncate_title()` 截断到 1200 字符 + `…` 后缀。阈值定 1200 基于**实测数据驱动**：扫描本地 2559 个 cached raw.md / 500856 个真 heading（已过 pseudo-heading filter），真最长合法标题为 **1137 字符**（来自 23.502 / 33.220 procedure spec 的 step 标题，如 `14a-c. 回归：真 spec 38.331 复跑 section_title **>200 chars: 1639 字符孤例 → 0**；7 个合法长标题（≤ 300）保留未截断。新增单测：
- `tests/unit/test_markdown_parser.py::TestPseudoHeadingAndLongTitle::test_skip_pseudo_heading_*`（3 个）
- `tests/unit/test_markdown_parser.py::TestPseudoHeadingAndLongTitle::test_long_title_truncated`
- `tests/unit/test_markdown_parser.py::TestPseudoHeadingAndLongTitle::test_legitimate_long_title_preserved`（验 38.331 §5.3.13.6 真标题不被误截）

## 10. 给后人的笔记

### 10.1 vision 的 image_path 必须是完整 HF repo_path

`ingestion/chunker/builder.py::_resolve_image_repo_path` 是关键修复点：markdown 中 `![alt](xxx_img.jpg)` 只携带 basename，**必须拼上** `Path(entry.raw_md_path).parent` 才是 HF dataset 完整路径（如 `marked/Rel-19/38_series/38331/xxx_img.jpg`）。

### 10.2 chunker 的 chunk_id 内容稳定但**不保证全 unique**

`chunk_id = uuid5(spec_id + clause + sha256(content)[:16])` 在同 section 内若 content 完全相同会产生同 ID。indexer 入口 `_dedupe_chunks` 保留 first occurrence。这是 belt-and-braces，不要因为 chunker 修了就删 dedupe（保留兜底）。

### 10.3 Voyage rate limit + free token 规则

- voyage **series 3**（voyage-3 / voyage-3-large）：200M free tokens（无需 payment method 也能用，但 rate limit 仍受限）
- voyage **series 4**（voyage-4 / voyage-4-large）：**没有 free token**，必须先 Voyage dashboard 加 payment method
- 没加 payment 时全 series rate limit：3 RPM / 10K TPM（38.331 单 spec 跑不通）
- 加了 payment 后 standard rate limit（2000 RPM / 1M TPM 量级）

### 10.4 PG schema 字段长度不要乐观估计

- `title` / `section_title` 用 `Text`（unbounded），别用 `VARCHAR(N)`：3GPP markdown 有 1639 字符的 section_title 边缘 case
- 真正应该限长的是 ID 字段（`chunk_id / parent_section_id / spec_id` 等）

### 10.5 voyage-4-large 默认 dim = 1024，不是 2048

docs / handoff 中提到 "voyage-4-large=2048" 是 Voyage 文档可选维度（输出维度可在 256/512/1024/2048 中选），**LiteLLM 默认 1024**。若要改 2048：

```yaml
# litellm config.yaml
- model_name: voyage-4-large
  litellm_params:
    model: voyage-ai/voyage-4-large
    output_dimension: 2048
```

然后 drop Qdrant collection 重建（dim 不可改）。

### 10.6 BM25 dedupe 在 ingestion 端做，不要让 backend 兜底

`BM25Writer.write_spec_chunks` 当前**不**做 chunk_id dedupe，直接 append。如果 chunker 给了 9042 chunks 含 189 重复 → BM25 source jsonl 也有 189 重复行 → backend build BM25Retriever 时会把同 chunk_id 索引两次 → 召回时返回同一 chunk_id 两次。

**修复**：在 `BM25Writer.write_spec_chunks` 入口加 dedupe（或在 pipeline 层把 dedupe 后的 chunks 传给所有 sink）。本次手工 dedupe 了 `by_spec/38.331.jsonl` 再 finalize，未来批量跑需要在代码层加。

### 10.7 retrieval smoke 验证 vision description 真的有用

Q1 ("RRC state machine transitions") top-2 都是 figure chunk（4.2.1 状态图），score 0.729 / 0.697。如果 vision description 没注入 content（M1 vision_resolver=None 的 fallback），figure chunk 只有 GSMA caption，retrieval 质量会大幅下降。**Vision pipeline 的价值在 retrieval 这里量化**。
