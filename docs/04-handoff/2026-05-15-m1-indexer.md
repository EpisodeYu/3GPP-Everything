## 3. 模块设计

### 3.1 `embedder.py`

| 公共符号 | 用途 |
|---------|------|
| `Embedder` | 主入口，按 provider + model 调 LiteLLM `/embeddings` |
| `EmbeddingError` | 对外异常 |
| `embed_chunks(emb, chunks)` | 抽 `content` 字段 → embed_texts 的便捷封装 |
| `PROVIDER_DEFAULT_MODEL` | `voyage` → `voyage-4-large` / `glm` → `embedding-3` |
| `_LiteLLMEmbeddingClient` | 薄封装 httpx 的 HTTP client；测试用 `_StubHttp` 替身 |

**关键逻辑**：

```
Embedder.embed_texts(texts: list[str])
  ├── 按 batch_size=64 切片
  ├── 每 batch 单独 retry：5xx / RequestError / TimeoutException 重试 3 次（tenacity 指数退避 1-30s）
  ├── 4xx → 立即抛 EmbeddingError（多半是 schema 错，重试无意义）
  ├── 按 response data[].index 重新排序（LiteLLM 偶尔乱序返回）
  ├── 校验 batch 内 dim 一致 + 跨 batch dim 一致（防 model 跑偏）
  └── 累计 prompt_tokens 用于成本统计
```

**dim 动态探测**：`Embedder.warmup()` 调一次 dummy embedding；首次 `embed_texts` 也会顺带探测。
QdrantWriter 据此建 collection。**项目标准向量维度 = 2048**（voyage-4-large 默认 1024，已在 LiteLLM `config.yaml` 显式声明 `output_dimension: 2048` 拉齐到 embedding-3 的 2048；voyage-3-large 1024 仅作对照，需要时单独开 collection）。

### 3.2 `qdrant_writer.py`

| 公共符号 | 用途 |
|---------|------|
| `QdrantWriter` | collection 创建 + payload index + upsert + spec-level purge |
| `collection_name_for_provider("voyage")` | `tgpp_chunks_voyage`（按 `QDRANT_COLLECTION_PREFIX`） |
| `DEFAULT_PAYLOAD_INDEXED_FIELDS` | `spec_id / spec_number / release / series / clause / chunk_type / parent_section_id` |

**幂等点**：
- `ensure_collection` 若已存在 → 跳过；dim 不会被改（防误操作）
- `create_payload_index` 重复调 → 吞 "already exists" 错（payload index 是 Qdrant 服务端 idempotent，但 client 偶报）
- `upsert_chunks` 用 `PointStruct(id=chunk_id)` → 同 id 替换不重复
- `purge_spec(spec_id)` 先 count 再 delete by FieldCondition

**payload 字段**（写入 Qdrant，召回时直接读避免 PG 查询）：见 `_chunk_to_payload`，含 content + raw_extra（含 vision JSON）。

### 3.3 `bm25_writer.py`

文件结构：

```
{INGEST_DATA_DIR}/bm25/{provider}/
├── chunks.jsonl       # 合并文件，每行 {"chunk_id","spec_id","clause","content",...}
├── meta.json          # {provider, total_chunks, spec_count, by_spec, written_at}
└── by_spec/
    └── {spec_id}.jsonl   # 单 spec 文件，覆盖写
```

**为什么这种结构**：
- 单 spec 重跑 → 直接覆盖 `by_spec/{spec_id}.jsonl`，幂等
- spec 级 purge → 删 `by_spec/{spec_id}.jsonl`
- finalize → 按 spec_id 字典序合并到 `chunks.jsonl`
- backend 启动时读 `chunks.jsonl` 现场 build BM25Retriever；不需要解读 LlamaIndex 矩阵

### 3.4 `pg_writer.py`

SQLAlchemy 2.0 Core（不用 ORM，避免 ingestion 端引入完整 backend ORM 上下文）：

```python
chunks_meta_table = Table(
    "chunks_meta", _metadata,
    Column("id", Integer, primary_key=True),
    Column("chunk_id", String(64), nullable=False, index=True),
    Column("provider", String(32), nullable=False, index=True),
    Column("spec_id", String(32), nullable=False, index=True),
    ... 全字段 ...
    Column("char_offset_start", Integer, nullable=True),    # 与 backend §3.1 一致但允许空
    Column("char_offset_end", Integer, nullable=True),
    Column("raw_extra", JSON, nullable=False),
    Column("cross_refs", JSON, nullable=False),
    UniqueConstraint("chunk_id", "provider", name="uq_chunks_meta_chunk_id_provider"),
)
```

**upsert 实现**：DELETE WHERE `(chunk_id IN batch) AND provider=X` + INSERT VALUES。
跨方言（PG + SQLite）都跑得起来，不依赖 PG-only 的 `ON CONFLICT`。

**dsn 转换**：`_to_sync_psycopg` 把 backend 的 `postgresql+asyncpg://...` 自动转 `postgresql+psycopg://...`（asyncpg 是 backend 的 async driver，ingestion 同步代码用 psycopg3）。共用一份 `DATABASE_URL` 即可。

### 3.5 `pipeline.py`

```python
@dataclass
class IndexerComponents:
    embedder: Embedder
    qdrant: QdrantWriter
    bm25: BM25Writer
    pg: PgChunkMetaWriter | None
    vision_resolver: ChunkerVisionResolver | None

def index_spec(bundle, components, *, chunk_params, purge_before=True) -> IndexStats: ...
def index_specs(bundles, components, *, skip_indexed, purge_before, progress_cb) -> PipelineStats: ...
```

单 spec 工作流：

```
build_chunks (chunker + vision)
  → embed_chunks (LiteLLM /embeddings)
  → qdrant.ensure_collection (dim 已探测)
  → purge_spec (Qdrant + BM25 + PG)  [if purge_before]
  → qdrant.upsert_chunks
  → bm25.write_spec_chunks
  → pg.upsert_chunks
  → return IndexStats
```

任一步抛异常 → 整 spec 视为失败，IndexStats.error 非空；`index_specs` 上层 continue 下一篇，不阻塞全量任务。

### 3.6 `runner.py`（CLI）

| 子命令 | 用途 |
|--------|------|
| `embed <spec_id>` | 单 spec chunker + embedding dry-run（不写 Qdrant/BM25/PG）；验证 LiteLLM 连通 + 估算 token |
| `index <spec_id>` | 单 spec 完整 indexer；可选 `--no-vision` / `--skip-pg` / `--no-purge-before` |
| `pipeline-hf` | 多 spec 批跑；`--spec-ids X,Y` / `--limit N` / `--only-whitelist` / `--skip-indexed` |
| `index-status` | Qdrant + BM25 + PG 状态查询；可按 spec_id 过滤 |
| `purge-spec <spec_id>` | 清掉单 spec 在 Qdrant + BM25 + PG 三处的所有写入 |

## 4. 关键决策记录

### 4.1 PG `chunks_meta` schema 所有权（过渡方案）

**背景**：backend alembic 未上线（M1 当前），但 indexer 必须写 `chunks_meta`。

**决策**：ingestion 端用 SQLAlchemy Core + `CREATE TABLE IF NOT EXISTS` 维护 chunks_meta 一张表，
字段以 docs §3.1 + chunker 实际输出最小子集为准。**不擅自建 documents / document_versions 两张表**
（属 backend 领地，§5.1 红线）。

**字段偏离 docs §3.1**：
- `document_id` FK 不创建（documents 表还没有）
- `char_offset_start` / `char_offset_end` nullable（chunker small2big 不产生准确偏移）
- chunk_type 用 `String(32)` 而非 `Enum`（chunker 实际输出 7 种 vs docs 列 4 种）
- UNIQUE 约束改为 `(chunk_id, provider)`（支持双轨 voyage + glm 共存）
- 新增 `spec_uid / spec_number / spec_type / release / series / title / clause / section_path /
  parent_section_id / parent_section_chars / document_order / cross_refs / source / source_version /
  created_at` 字段（chunker 实际输出，便于 PG-only 查询不依赖 Qdrant）

**移交计划**：backend alembic 上线后，第一份 revision 用 `op.create_table_if_not_exists` 接管。
schema 完全一致 → 无迁移痛苦；若 backend 想加 `Enum chunk_type` / `documents FK`，走 alembic
后续 revision 即可（ingestion 同步更新 pg_writer.py 即可）。

### 4.2 BM25 持久化 = "源数据" 而非 "索引矩阵"

**背景**：docs §4.4 "BM25：LlamaIndex BM25Retriever，全量重建（50k+ chunks < 60s）"。

**两条候选路径**：

| 方案 | 优 | 劣 |
|------|----|----|
| **A. ingestion 直接持久化 LlamaIndex BM25Retriever** | backend 启动即用，零构建时间 | ingestion 要加 `llama-index>=0.13`、`llama-index-retrievers-bm25>=0.4` 等重依赖；spec 级 purge 难做 |
| **B（采用）. ingestion 只写 chunks.jsonl，backend 加载时现场 build** | ingestion 依赖纯净；spec 级 purge 简单（删 by_spec/ 单文件）；50k 现场 build < 60s 完全可接受 | backend 启动慢 < 60s 一次性成本 |

**选 B**。文档同步把"持久化"定义为"持久化 BM25 的源 chunks"。

### 4.3 向量 dim 动态探测

voyage-4-large(LiteLLM 显式配 2048) / voyage-3-large(1024) / embedding-3(2048)。`.env` 仅给 model name 不给 dim；项目标准向量维度 = **2048**。
QdrantWriter `ensure_collection(dim=...)` 必须知道 dim。两种实现：

- 硬编码 `dim_per_model = {...}` —— 模型新版本来了就要改
- **首次 embed 调用拿 len(vector[0]) 推断** —— 零配置

选后者。代价是 `index_spec` 内 embedding 必须先于 `ensure_collection`；CLI 也支持 `embedder.warmup()`。

### 4.4 续传 / 重跑策略

不引入 PG `documents.status` 状态机（docs §4.6 提到但属于 backend alembic 范畴）。**只靠数据本身的幂等性**：

- chunk_id = uuid5(spec_id|clause|sha256(content)[:16])：内容稳定 → ID 稳定 → upsert 不重复
- `--purge-before` 写之前清旧 ID（防止 vision/chunker 改了内容后 ID 漂移导致脏记录）
- `--skip-indexed` 查 Qdrant point > 0 判断（增量场景）

后续 M6 全量索引时如有需要再上 PG `documents.status` 状态表（backend 领地）。

### 4.5 同步 vs 异步

ingestion 全栈同步：httpx.Client / SQLAlchemy 同步 / Qdrant 同步。
理由：indexer 是命令行批跑工具，**bulk-throughput** 远比 latency 重要；同步代码可读性 + 调试性碾压 async。
backend 是 FastAPI + asyncpg + LangGraph async checkpointer 异步栈，与 ingestion 解耦。

## 6. 重要注意点 / 坑

### 6.1 LiteLLM proxy 必须配置（dim 探测时一定要打）

`embed` / `index` / `pipeline-hf` 启动时都要打一次真实 LiteLLM。本地宿主跑必须：

```bash
export LITELLM_BASE_URL=http://localhost:4000/v1
export LITELLM_API_KEY=...
```

容器内跑保留 `.env` 默认 `http://host.docker.internal:4000/v1` 即可。

### 6.2 Qdrant collection dim 不可改

`ensure_collection` 若 collection 已存在 → **不会检查 dim 是否一致**（防误操作）。
换模型（voyage-4-large → voyage-3-large）必须：

```bash
# 显式 drop 旧 collection（qdrant-client）后重跑
curl -X DELETE "$QDRANT_URL/collections/tgpp_chunks_voyage"
ingestion index 38.331 --provider voyage
```

未来可以加 `--rebuild-collection` 选项，但属 M3/M6 关切。

### 6.3 `purge_before=True` 是强幂等的代价

每次 `index` 单 spec 都会先调 `qdrant.purge_spec()` 把旧 point 全删 → upsert 全新内容。
单 spec ~3000 chunks 时 delete + insert ≈ 1-2s（PG 同理）。批跑 1296 篇 ≈ 30 min。
**不接受**这个代价的场景（如 M6 后单 spec 增量补丁）请 `--no-purge-before`，靠 chunk_id uuid5
的 content-stability 自然去重（缺点：若 vision 改了 description 但部分 chunk 没变 → 旧 chunk 不会被删，需另查另删）。

### 6.4 PG schema 由 ingestion 维护是过渡方案

详见 §4.1。**backend alembic 上线时必须做**：
1. 在 backend `alembic/versions/0001_init.py` 用 `op.create_table_if_not_exists` 包住
2. 字段名 / 类型 / 索引 / 唯一约束完全对齐 ingestion 的 `chunks_meta_table` 定义
3. 把 `pg_writer.py` 顶部注释中的"过渡方案"段移除，标注 backend 已接管

### 6.5 BM25 chunks.jsonl 必须 finalize 后才能给 backend 用

单 spec 跑完 `index` → 只写了 `by_spec/{spec_id}.jsonl`，**不会**自动更新 `chunks.jsonl`。
`pipeline-hf` 默认 `finalize_bm25=True` 跑完所有 spec 后才合并。
手动单 spec 后需要 backend 立即生效请：

```python
from ingestion.indexer import BM25Writer
BM25Writer(provider="voyage").finalize()
```

或加一个 `ingestion bm25-finalize` 子命令（M2 实施时按需补）。

### 6.6 双 provider 共存的成本

`(chunk_id, provider)` UNIQUE 设计允许 voyage + glm 同 chunk_id 共存于 PG。
单 spec 跑两次（voyage 一次，glm 一次）→ PG 行数翻倍、Qdrant 两 collection 各一份。
**POC 期 docs §5 已警告："默认串行跑、跑完即清"**。M2 评测期跑完一个 provider 立刻
`ingestion pipeline-hf --provider X --purge-spec` 或直接 drop collection + DELETE FROM
`chunks_meta WHERE provider=X` 清掉。

### 6.7 单测用 SQLite + in-memory Qdrant，**不**打真实服务

47 项 indexer 单测全部 < 6 秒跑完，0 网络 I/O。**绝对不要**改成"连真实 Qdrant / PG"——慢、贵、CI 会炸。
真实集成测试由 M2 POC `index 38.331` 端到端跑当担。

### 6.8 Embedder retry 不重试 4xx

mimo / voyage / glm 任一返回 400/401/403/422 → 立即抛 EmbeddingError，不走 tenacity。
理由：4xx 多半是 model name 错、API key 失效、token 限额、payload 超长——重试 3 次只会重复同错误。
排查思路：先 `ingestion embed <spec_id> --batch-size 1 --no-vision` 单文本调用确认链路。

## 10. 给后人的笔记

- **chunk_id 是内容稳定的 uuid5**：plan §3 的核心保证。改了 chunk 内容 → ID 变；
  保留 `purge_before=True` 才能确保旧 ID 被清。不要为了"省时间"默认关掉 purge_before。
- **provider 不只是文件名后缀**：voyage / glm 在 Qdrant collection 名、PG provider 列、
  BM25 子目录三处都做了隔离。新增 provider（如 jina / openai）只需扩 `Embedder.PROVIDER_DEFAULT_MODEL`
  + 新增 model env，其余代码不改。
- **向量 dim 探测一次永久缓存**：`Embedder._dim` 跨 batch 校验一致；运行中改 model name
  会被检测到并抛错。生产改 model 必须重启进程 + drop collection（见 §6.2）。
- **失败的 spec 不该阻塞全量**：`index_specs` 单 spec catch 后 continue，stats 里 `failures: list`
  累积所有失败。CLI 输出 `[pipeline-hf] failures: ...` 给运维一目了然。
- **PG UPSERT 用 DELETE+INSERT 不用 ON CONFLICT**：跨方言（PG + SQLite）兼容是单测能在
  SQLite 跑的关键。代价是大批量 spec 重跑会有少量"先删后插"的窗口期 race condition——
  单 spec 重跑期间不可被其他 worker 同时索引同一 spec（生产串行无问题）。
- **BM25 不是 LlamaIndex 索引**：当前持久化的只是 chunks 源数据。backend 加载时构建 BM25
  矩阵 < 60s（5w chunks），完全 OK。未来想做"持久化矩阵 + 增量更新"是 M3+ 性能优化范畴。
- **单测刻意只用 in-memory**：47 项 < 6s 跑完。改成"连真实 PG + Qdrant"会慢一个量级
  且 CI 配置复杂。M2 起的"真实链路 smoke"由集成测试 / 手跑 `index 38.331` 担纲。
