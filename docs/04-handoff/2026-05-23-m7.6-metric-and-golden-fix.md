## 1. 输入分析

读 `eval-results/m7-rerank-ablation.md`（M7.5 基线，2026-05-22）。56 题
hand_crafted 子集，40 题进入召回。各 config 失败题清单共有 9 题被列为 fail，
其中 4 题 (`hand-def-002` / `hand-multi-001` / `hand-table-001` / `hand-formula-002`
/ `hand-multi-004` 等) 在多个 config 下都 0 hit。

按"chunk 真实存在性 + 真实协议对照 + retrieval 行为"三个维度做了诊断（不只读
报告，连协议章节都查 PG / 本地 `bm25/by_spec` 的真实 chunk 内容），发现 **三类
根因**：

### 1.1 根因 A — eval metric 没有 merged-chunk 感知（最高 ROI）

`ingestion/chunker` 把小相邻子节合并到父级 chunk 时，会让 `section_title` 形如
`<merged: 4.4.4.3 / 4.4.4.4 / 4.4.4.5>`，对应的 `clause` 留在父级（如 `4.4.4`），
chunk 内容里实际包含子节正文（验证：直接读
`/data/tgpp/bm25/voyage/by_spec/38.211.jsonl` 里的 `clause="4.4.4"` 记录，content
含 `#### 4.4.4.3 Common resource blocks` 全文）。

M7.5 的 `is_section_hit` 只走 `is_section_prefix(expected, hit.section_path)` 一个
方向，遇到 expected=4.4.4.3 + chunk@4.4.4 时 `len(["4","4","4","3"]) > len(["4","4","4"])`
判 miss。涉及失败题：

- `hand-formula-002`（expected 38.211 §4.4.4.3，merged 进 §4.4.4）
- `hand-table-001`（expected 38.211 §4.2，merged 进 §4，title `<merged: 4.1 / 4.2>`）
- `hand-multi-004`（expected ... §6.3.2.5.1 / §6.3.2.5.2，merged 进 §6.3.2.5）

### 1.2 根因 B — golden hand-multi-004 spec_id 写错

问 PUCCH format 2 的 Scrambling + Modulation。原标注 `38.212 §6.3.2.5.1 /
§6.3.2.5.2`，但：

- 38.212 = Multiplexing & channel coding，**整章没有 §6.3.2**（直接查 PG
  `chunks_meta`：`SELECT … WHERE spec_id='38.212' AND clause LIKE '6.3.2%'` 返
  回 0 行）
- PUCCH format 2 物理通道处理（Scrambling / Modulation / Mapping）实际在
  **38.211 §6.3.2.5.\***（验证：PG 查出 `clause='6.3.2.5' title='<merged:
  6.3.2.5.1 / 6.3.2.5.2>'`，content 第一行 `[38.211 § 6.3.2.5 ...]` + 子节正文）

### 1.3 根因 C — 5G NR 题被 LTE 36.\* 系列同名 chunk 抢答（残留主问题）

修完 A+B 之后的 6 个 wide_dense_rerank5 fail 里，过半数 top-3 hit 都落在 36.211 /
36.212 / 36.521-3：

- `hand-multi-001`（PDSCH 调制+layer mapping §7.3.1.2/3）→ top-3 全 36.211
- `hand-multi-004`（PUCCH format 2）→ 36.211 §5.4.2 抢答
- `hand-proc-005`（PRACH transmission power §7.4）→ 36.521-3 §6.2.\* 抢答

PUCCH / PDSCH / PRACH 在 LTE 与 NR 里概念名一致；Voyage embedding + rerank-2.5
仅从 chunk content 内的"38.211" 自身 spec id 标签无法可靠区分。

### 1.4 其他 retrieval miss

- `hand-def-002` 问 "什么是 BWP"。chunk content 用 "bandwidth part"，从不
  写 "BWP"，被 38.331 RRC 配置项里铺天盖地的 "bwp-Id / bwp-Common" 噪声压住。
- `hand-proc-001` 问 38.211 §6.3.1.1 PUSCH Scrambling。retrieval top-1 是 38.211
  §6.3.2.6.1（PUCCH Scrambling）—— **相邻同名节互相挤压**，与同 spec 内其他
  Scrambling 节区分度低。
- `hand-proc-003` 问 38.212 §7.3.1.0 DCI size alignment。top-3 全是 38.212 其他
  DCI 子节（spec 对了，section 错了）。
- `hand-table-004` 问 38.212 §5.4.1.1 Polar subblock interleaver。top-3 全是相邻
  §5.3.1.2。

## 5. 留给后续 / M8 的 tech-debt

按 ROI 排序：

### 5.1 LTE / NR 同名概念区分（最大头）

**症状**：5 个失败里 3 个被 36.\* 抢。`hand_crafted` 子集 100% 是 5G NR，但
retrieval / rerank 没办法只从短 query 判定 NR vs LTE 上下文。

**候选方案**（按代价从低到高）：

1. **classify 节点感知 spec_series**：classify 输出多加一个 `target_spec_series`
   字段（`"38.*" | "36.*" | "23.*" | "29.*" | "any"`），在 dense `retrieve` 阶
   段把它转 `qmodels.FieldCondition(key='spec_id', match=qmodels.MatchAny([...]))`
   过滤。
2. **chunk content ingest-side enrichment**：在每条 chunk 前缀里显式拼出 spec
   所属 generation（`[5G NR]` / `[LTE]` / `[5GS core]`），让 voyage embedding +
   rerank 学到 generation 级别的语义差异。
3. **multi-query rewrite** 让 LLM 直接把生成代次写进 sub-query（生产已有
   `nodes/multi_query.py`，但 ablation 脚本绕过了）。

### 5.2 缩写词典在 ingest-side 注入（中等优先级）

`hand-def-002` 问 BWP 命中不到 38.211 §4.4.5。原因是 chunk 只有 "bandwidth
part" 而 query 只有 "BWP"。Voyage 多语种 embedding 应该能桥但实测不够。方案：

- 用 `ingestion/glossary/` 已有的缩写表，在 chunk 末尾追加 `[术语：BWP =
  bandwidth part; PDU = ...]` 这种 "expansion footer"，让 BM25 命中 BWP token、
  dense 也得到双语种锚点。
- 风险：所有 chunk 都被加 footer 会推高 embedding token 成本（394k chunks ×
  ~40 token footer ≈ 15M token 增量，需要 ≥ 5M token approval）。

### 5.3 chunker merge 行为（可选）

现在 merge 后 chunk 把多个子节内容塞一起、`clause` 只留父级。这让"chunk 真实
覆盖范围"和"`clause` 字段"不一致，本次靠 metric 解析 title 兜底。更彻底的做
法是 chunker 输出 `coverage_clauses: list[str]`（既保留 `clause` 主字段，又显
式记录所有被合并的子节 id），然后 metric 直接用结构化字段判定。需要 reindex
所有 chunks_meta（~17M 条），代价较高，不紧急。

### 5.4 同 spec 内相邻同名节区分（低优先级，影响题数少）

`hand-proc-001` PUSCH Scrambling vs PUCCH Scrambling，`hand-table-004` Polar
subblock interleaver 被相邻 §5.3.1.2 挤掉。靠 rerank-2.5 当下无法区分，可能要
等 cross-encoder rerank（如 BGE-Rerank-v3）或 LLM-rerank 引入。
