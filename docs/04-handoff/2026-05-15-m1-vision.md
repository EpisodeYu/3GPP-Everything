## 3. 模块设计

### 3.1 `prompts.py`

| 公共符号 | 用途 |
|---------|------|
| `PROMPT_E_UNIFIED` | 与 docs §4.2.1 一字一句对齐；vision.py 与 mini benchmark 都引用同一份 |
| `VALID_FIGURE_KINDS` | 11 种合法 figure_kind 枚举集 |
| `parse_vision_json(text)` | 容错解析 mimo 响应：原文 → 围栏剥离 → 截首个 `{...}` 块 |
| `normalize_vision_payload(payload)` | 对齐 §4.2.2 schema：缺字段补默认 / figure_kind 校验 / 空 description → None |

**为什么独立成文件**：mini benchmark 脚本 / vision.py / 未来 evaluator 都要引用 prompt
与解析工具；与 HTTP / Redis 调用解耦让单测更纯。

### 3.2 `vision.py`

```python
@dataclass(slots=True)
class VisionResult:
    description, figure_kind, visible_labels, visible_acronyms,
    spec_role, undescribable_reason,
    model, completion_tokens, reasoning_tokens, cached, raw_response

class VisionResolver:
    """符合 chunker figure.py::vision_resolver 的 callable。"""
    def __call__(self, image_path: str, ctx: dict) -> dict | None: ...
```

**关键流程**：

```
__call__(image_path, ctx)
  ├── _image_loader(image_path) → (bytes, sha256)
  ├── cache.is_dead(sha256) → return None  （不再调，跳过）
  ├── cache.get(sha256) → return cached_dict（cached=True）
  ├── for attempt in 0..max_retries:
  │     ├── call_mimo_unified(http, image_bytes, model, max_tokens) → (payload, meta)
  │     ├── 失败种类：HTTP / finish_reason=length / JSON 解析 / normalize 不通过
  │     ├── 失败 → cache.bump_retry(sha256, error, ctx_pruned)
  │     └── 成功 → return VisionResult
  ├── 全部尝试用尽 → cache.move_to_dead(sha256, error, ctx_pruned)
  │                 → on_dead_letter(image_path, ctx, error)
  │                 → raise VisionDeadLetterError → 外层 catch 返回 None
  ├── cache.set(sha256, result_dict)（节省 Redis：raw_response 不入缓存）
  └── return result_dict（cached=False）
```

**ctx 字段精简**：`_prune_ctx(ctx)` 只保留 `spec_id / clause / section_title / image_alt /
spec_caption`；`surrounding_paragraph` / `gsma_caption_text` 可能很长，进 Redis 浪费空间。

**Redis key 设计**（与 §4.2.3 对齐）：

| Key | 写入时机 | 含义 |
|-----|---------|------|
| `tgpp:vision:{sha256}` | 每次成功（含 undescribable） | description + 结构化字段；永久 TTL；体积 ~1KB |
| `tgpp:vision:retry:{sha256}` | 每次失败 | `{retry_count, last_error, last_at, ctx_pruned}`；成功或落 dead 时清除 |
| `tgpp:vision:dead:{sha256}` | 重试 3 次仍失败 | `{retry_count, last_error, moved_at, ctx_pruned}`；监控 / 排障入口 |

**注入点**（测试 / debug 友好）：

- `http_client`：`_LiteLLMClient` 替身（测试用 `_StubHttp`，按队列吐 payload）
- `cache`：`_VisionCache(redis_client=fakeredis.FakeRedis(...))`
- `image_loader`：`Callable[[str], (bytes, sha256)]`（测试直接吐 fixture bytes）
- `model` / `max_tokens` / `max_retries` / `image_mime` / `on_dead_letter`：覆盖默认

### 3.3 `runner.py`（CLI）

| 子命令 | 用途 |
|--------|------|
| `vision-call <image_path> [--no-cache] [--max-tokens N] [--max-retries N]` | 单图调用 + 人类友好排版（description / labels / acronyms） |
| `vision-call-json <image_path>` | 同上但纯 JSON 输出，便于 jq pipe |

## 6. 重要注意点 / 坑

### 6.1 .env 里 `host.docker.internal` 仅容器内可用

宿主机直接跑 vision.py / mini benchmark / CLI 必须 export 覆盖：

```bash
export LITELLM_BASE_URL=http://localhost:4000/v1
export REDIS_URL=redis://localhost:6379/5
```

容器里跑（M2 全量索引时）保留 `.env` 默认即可。CI 同理需要注入 service container 的真实 host。

### 6.2 mimo-v2.5 是 reasoning 模型，max_tokens=8192 是必要的

vision.py 默认 `DEFAULT_MAX_TOKENS=8192`，**不要**降到 4k。原因：

- mimo 的 reasoning_tokens 占用 max_tokens 配额（mini benchmark median rt ≈ 1.3k，max ~3.2k）
- 若被 max_tokens 截断，`finish_reason=length` 会让 vision.py 计入失败重试
- max_tokens 设大不增加成本（按实际 ct 计费），只占内存暂存空间

### 6.3 PROMPT_E_UNIFIED 与 docs §4.2.1 必须双向同步

`prompts.py` 文件头部的注释已写明：任何修改必须先在 mini benchmark 跑过 + 文档同步。
**不要**只改 prompt 不改文档，反之亦然。

### 6.4 Redis 不可用时 vision.py 仍能跑（cache disabled fallback）

`_VisionCache` 在 `REDIS_URL` 未配置或连接失败时，所有 `get` 返回 None，所有 `set` 是
no-op。CI 离线场景不会因为 Redis 不在而炸，但失去缓存能力（每次调用都打 mimo）。

### 6.5 `figure_kind=undescribable` 不算失败，照常缓存

避免反复调用一张已确认描述不出来的图。chunker 拿到 `undescribable_reason` 后可以决定是否
排除出 embedding（M3 评测期决策）。M1 fallback 行为：照常写 description（mimo 给的"1 句话"），
进 chunk content。

### 6.6 `_StubHttp` 不调父类 `__init__`

测试里 `_StubHttp` 继承 `_LiteLLMClient` 但跳过 httpx 客户端构造（不需要真 client）。
如果未来 `_LiteLLMClient.__init__` 加新必须字段，`_StubHttp.__init__` 要同步。
单测里 22 项是用 `_StubHttp.chat()` mock 走的，**不真打网络**。

### 6.7 ctx 在 retry / dead-letter 入 Redis 前 prune

`surrounding_paragraph` / `gsma_caption_text` 可能上千字符。`_prune_ctx` 只保留 5 个核心
字段（spec_id / clause / section_title / image_alt / spec_caption），dead-letter 排查用。

### 6.8 chunker 现在还没接 vision_resolver

M1 chunker（上一份 handoff）的 `figure.py::build_figure_content` 接口已经留好 `vision_resolver`
参数；`builder.py` 当前 `vision_resolver=None` 跑 GSMA 自带描述兜底（fallback 到方案 A）。
**M2 indexer 实施时**才把 `VisionResolver()` 实例传进去；chunker 代码本身不需要改。

### 6.9 chunker 重跑后 chunk_id 会变

chunk_id = uuid5(spec_id|clause|sha256(content)[:16])。M2 接入 vision.py 后：
- figure chunk content 会从"GSMA 描述"变为"mimo description + visible_labels + visible_acronyms"
- → content hash 变 → chunk_id 变
- Qdrant 要 upsert + purge 旧 figure chunks（按 spec_id × chunk_type=figure 选删）

handoff `2026-05-15-m1-chunker.md §8.2` 已点出这个问题。

## 10. 给后人的笔记

- **prompt 是契约**：PROMPT_E_UNIFIED 在 docs §4.2.1 与 prompts.py 双份维护是有意为之——
  docs 是给人审 / 设计变更追溯，代码是给运行。改一处必须同步另一处，否则 mini benchmark 会
  跑出与文档不一致的结果，污染未来的回归对比。
- **测试用 fakeredis + StubHttp 不打网络**：22 项单测全部 in-memory 跑 < 2 秒。**绝对不要**
  把测试改成"打真实 LiteLLM"——慢且贵，CI 会炸。真集成测试由 mini benchmark 担。
- **mini benchmark 是上线门禁，不是 CI 测试**：跑一次 ~5 分钟 + ~$0.30，CI 不跑（默认）；
  vision.py / prompts.py 任何改动后人工跑一次。结果落 `eval-results/source-audit/`，
  与 docs 一起 review。
- **failure 不止 HTTP 错**：JSON 解析失败 / `finish_reason=length` 都计入失败重试。这是
  在 mini benchmark 调试时实测发现：mimo 偶发输出"Sure, here is the JSON: \n{...}"，
  parse_vision_json 的"截首个 {...} 块"兜底正好接住。
- **dead-letter 比 retry 更重要**：retry 队列堆积只是慢，dead-letter 是"明确放弃"，
  必须有人值守清理。M2 indexer audit 必须出"本次新增 dead-letter 数"指标。
- **环境差异是真实问题**：本次第一次跑 mini benchmark 因 `host.docker.internal` 在宿主机
  不解析全失败。这种坑只有真跑才发现，不会被 unit test 兜住。**写脚本时尽量明确"在哪种
  环境跑"**（容器内 vs 宿主机），README 标清楚 export 哪些 env。
