## 2. 设计决策

### 2.1 merge：collision 默认 fail，`--force` 后赢

`force_overlap=False`（默认）→ 任何跨文件 id 冲突即 error，**不写文件**；error 同时
给出冲突双方 `file:line`。理由：合并是"把人审过的手写题 cat 进 teleqna 主集"，静默
覆盖等于丢人审结果。`--force` 留给"故意改 teleqna 转化的某题"场景（罕见）。

### 2.2 stats：硬编码 §3.4 目标 + tolerance 默认 5

`CATEGORY_TARGETS` 是文档 §3.4 的镜像（30/35/10/10/10/10/15 = 120）。
`tolerance=5` 是 §0 写明的容差。category 目标改 → 同步改文档 + 这张表。

`SOURCE_TARGETS['hand_crafted']=20` 是 §0 决策 Q1 落实：daily eval 切片 ≥ 20 题。
`teleqna_transformed` 无目标，列 INFO 行不阻塞。

### 2.3 SOP：写在文档而不是 Makefile

放 `06-...md §3.6` 而不是 `Makefile` 目标的原因：
- 8 步里 1-3 涉及外部依赖（GitHub clone / LLM 调用 / 人审），不能简单 `make`
- 每步独立可重跑，文档给清楚每步的输入 / 输出 / 何时跑，比 phony target 更有用
- M7.6 真要做 CI 时再把"步骤 5 + 8"封装成 `make eval-golden-check`
