## 1. 起因：连续 3 个症状逐层揭出 4 道关

人在 Cloud UI 配 evaluator 时连续踩坑：

| 顺序 | 症状 | 根因 |
|---|---|---|
| 1 | Dataset `tgpp-golden-v1` 页面找不到"启用 evaluator"按钮 | M7.3 §5.2 文档不准；evaluator 在独立 `Evaluators` 页面配 |
| 2 | LLM Connection 保存失败 `score: expected string, received number` | 国内模型（mimo）structured output 不严格 |
| 3 | 换 OpenAI base URL → `Unexpected token '<'`(HTML) | 鉴权 header 不对（mimo 用 `api-key` 而非 `Authorization: Bearer`） |
| 4 | 换 DeepSeek（OpenAI adapter） → `400 This response_format type is unavailable now` | DeepSeek/GLM/Qwen/Moonshot/Mimo 都不支持 OpenAI strict `response_format: json_schema` |

排查到第 4 步发现"国内 OpenAI 兼容端点几乎全军覆没"——Langfuse evaluator 在
保存模型时下发的是 OpenAI 2024-08-06 引入的严格 schema 模式，国内厂商 2026-05
时点还没普遍跟进。最终走通的是 **DeepSeek 自家的 Anthropic 兼容端点 + Langfuse
`adapter=anthropic`**（Anthropic 用 tool use 实现 strict structured output，与
OpenAI 的 response_format 是两套机制）。

**但即便 connection 通了，runner 还有 4 个隐性硬阻塞**（fixup-1 / fixup-2 两轮修完）：

- **关 2** [fixup-1]：runner 只调 `create_event` + `create_score`，**没调
  `dataset_run_items.create`** → trace 不挂 dataset run，UI Datasets→Runs 子页面看不到。
- **关 3** [fixup-1]：trace `output` 只有 `{answer, terminal_event, citations}`，
  **没有 `contexts` 字段** → faithfulness 评估器读 `{{context}}` 拿到空字符串。
- **关 4** [fixup-2]：runner 用 `create_event` 创建 root observation 是 **EVENT 类型**，
  且 trace **没有 5 个 experiment OTel span attributes**（`langfuse.experiment.id` /
  `dataset.id` / `name` / `item.id` / `item.root_observation_id`）也没设置
  `langfuse.environment=sdk-experiment` → v3.175 Cloud LLM-as-a-Judge evaluator 的
  filter `experimentDatasetId any of <id>` **就是读这些 OTel attribute**，
  全部缺失 → evaluator 永远不派发，UI Experiments 标签页看不到，
  `langfuse-llm-as-a-judge` 环境下也没有任何 evaluator 执行 trace。
- **关 5** [fixup-2]：v3.175.0 Cloud 没有"backfill 历史 dataset run"按钮，
  evaluator 配错过的 dataset run 无法回填 → 修完关 4 后必须**触发新 run** 才能看到 score。
- **关 6** [fixup-3]：backend SSE `chunks_rerank/chunks_hit` 事件每 chunk 只发
  240 字 `preview`（前端展示用），完整 chunk text 不发 → runner `_join_contexts`
  能拼到的 evidence 严重缺失 → faithfulness 评出来 2/3 题 0.0 全是 false negative
  （fixup-2 实测：`6baa9eb6=0.727 / 12e96a1a=0.0 / 96ed337a=0.0`）。修法：
  backend 同时发 `content`（完整文本）+ `preview`（前端兼容），runner 优先取 content。

## 2. 修补内容

### 2.1 fixup-1（上午）：`eval/runner.py`

**新增**：

- `_join_contexts(resp, *, max_chars=8000) -> str`：
  - 拼 `chunks_rerank`（fallback `chunks_hit`）的 `preview`（240 字／chunk，
    backend SSE 的最大可见量；完整 chunk text 不发 SSE，按 §7 "不重启 backend"
    约束 runner 也不去 Qdrant 重查）；fallback `text` / `content` 字段防御未来 schema 变更
  - 每 chunk 拼成 `[<spec_id> §<section_path>]\n<preview>`，多 chunk 用
    `\n\n---\n\n` 分隔
  - `max_chars` 截断防 evaluator judge 上下文爆掉
- `_link_trace_to_dataset_run(lf_client, *, trace_id, dataset_name, dataset_item_id, run_label) -> bool`：
  - 调 v4 低层 REST `lf_client.api.dataset_run_items.create(run_name, dataset_item_id, trace_id, metadata)`
  - `dataset_name=None`（runner 没传）→ 跳过返回 False（孤儿 run 也是合法回退路径）
  - 任何异常吞 + log + 返回 False

### 2.2 fixup-2（下午）：`eval/runner.py` 关键改动

**根因**（用 SDK 反查锁定）：v3.175 Cloud LLM-as-a-Judge 的 evaluator filter
`experimentDatasetId any of <id>` 不是从数据库 `dataset_run_items` 表反向 join，
而是直接读 trace 的 OTel span attribute `langfuse.experiment.dataset.id`。
SDK `dataset.run_experiment(task=...)` 内部（`langfuse/_client/client.py:2780`）
就是这么干的：先 `start_as_current_observation(as_type='span')` 创建 root span，
再 `_otel_span.set_attributes(EXPERIMENT_*)`。

**`_emit_langfuse_trace_event` 重写为 `_emit_langfuse_experiment_span`**：

- 用 `lf_client.start_as_current_observation(name=..., as_type="span", input=..., output=..., metadata=...)`
  上下文管理器（不再用 `create_event`）
- 在 span 上 `set_attributes` 5 个 experiment OTel attribute：
  - `langfuse.environment` = `"sdk-experiment"`（特殊值，与 SDK 常量
    `LANGFUSE_SDK_EXPERIMENT_ENVIRONMENT` 一致）
  - `langfuse.experiment.id` = `run_label`
  - `langfuse.experiment.name` = `run_label`
  - `langfuse.experiment.dataset.id` = lookup 出来的 dataset 内部 ID
  - `langfuse.experiment.item.id` = `item.id`
  - `langfuse.experiment.item.root_observation_id` = span.id
- 返回 `(trace_id, observation_id)`；trace_id 由 OTel span 自动生成（不再依赖
  `make_eval_trace_id` 的 seed）

**新增 `_resolve_dataset_id(lf_client, dataset_name) -> str | None`**：

- 一次性 `lf_client.api.datasets.get(dataset_name=...)` 拿 dataset 内部 ID（OTel attribute
  要 ID 不要 name）
- 缺 client / dataset_name / 异常 → 返回 None；caller 不会 set EXPERIMENT_DATASET_ID，
  evaluator 仍能跑但 dataset filter 不命中（优雅降级）

**`run_eval` 调用链改造**：

- 启动时调一次 `_resolve_dataset_id` 缓存到本地 `lf_dataset_id`
- 循环里：`_emit_langfuse_experiment_span(...)` → `_link_trace_to_dataset_run(...)` →
  `push_run_score(...)`；不再调 `make_eval_trace_id`

**`make_eval_trace_id` 废弃但保留**（在 `eval/langfuse_dataset.py`），未来如有
"不挂 experiment 只想推 score"的回退场景仍可用；`__all__` 里相应替换。

### 2.5 fixup-3（收尾）：扩 contexts 信息量解决 false-negative

**问题**：fixup-2 跑完后 evaluator 真的派发了，但分数偏低（2/3 题 0.0）。原因是
runner 拼 `trace.output.contexts` 只能拿到 backend SSE `preview` 字段（240 字），
top-K=10 时总量也才 ~1100-1300 字符；很多 chunk 的关键 evidence 在 240 字之后被截掉，
evaluator 自然判 unfaithful。

**修补**（M7.3 主体的"不重启 backend"约束已在 M7.2 评估结束后解除）：

#### `backend/app/agent/nodes/rerank.py` + `retrieve.py`

`_emit_chunks_rerank` / `_emit_chunks_hit` 的 chunk payload **新增 `content` 字段
= 完整 chunk 文本**；`preview`（240 字）保留给前端 UI 流式展示，向后兼容。

```python
# rerank.py & retrieve.py 同口径
{
    ...,
    "preview": (c.content or "")[:240],   # 前端用
    "content": c.content or "",           # eval runner 拼 contexts 用
}
```

字段是新增不是替换，前端零改动。`backend/tests/integration/agent/test_sse_events.py`
的 schema 断言加上 `content` key。

#### `eval/runner.py::_join_contexts`

- 字段优先级改为：`content > preview > text`（防御老 backend 部署）
- `max_chars` 默认 `8000` → `16000`（完整 chunk 后 top-K 总量变大；16K char ≈
  4-5K token，对 deepseek/gpt-4o-mini class 64K 上下文 < 10%，留足余量）

#### 单测

- `TestJoinContexts.test_field_priority_content_then_preview_then_text` 替换原
  `test_field_fallback_text_then_content`：assert content 优先级最高
- 新增 `test_uses_content_when_present`：full-content 拼接长度 > 800 字符
- `test_langfuse_dataset.py::_sse_body()` 补 `content` 字段

## 4. 自主决策记录（按 CLAUDE.md §4.3）

1. **`contexts` 用 SSE preview（240 字／chunk）拼，不去 Qdrant 重查**：M7.3 §7
   "不重启 backend"约束仍在；preview 对 faithfulness 已够（faithfulness 看
   answer 每个事实是否在 context 出现）；后续如发现 evaluator 偏差大，
   再按需扩 backend SSE schema 或在 runner 里查 Qdrant 补完整 chunk text。
2. **`max_chars=8000` 默认值**：约 4 个 1024-token chunk 的拼接量级。
3. **fixup-1 用低层 `dataset_run_items.create`，fixup-2 改用 `start_as_current_observation` +
   手动 set OTel attributes，仍不切到高层 `dataset.run_experiment(task=...)`**：
   - 保持现有 for 循环 + `push_run_score` 架构最小侵入
   - 不重写为"task function 模式"以免 ragas / negative_judge / SSE 消费链路全部要适配
   - 代价：依赖 SDK 私有句柄 `span._otel_span.set_attributes(...)`（SDK 自家 `run_experiment`
     也用同一套，且 OTel 标准 API 稳定，升级 langfuse 主版本时回归一次即可）
4. **OTel attribute key 用字面量常量化（不 import `LangfuseOtelSpanAttributes`）**：避免
   依赖 SDK 私有模块（`langfuse._client.attributes`），降低升版本爆炸面；常量在
   `runner.py` 顶部集中定义，注释里写明"SDK 升版本时 grep 校对一次"。
5. **`dataset_id` 启动时一次性 lookup 缓存复用**：175 题循环里每次查 dataset.id
   是浪费；用 `_resolve_dataset_id` 提到外面，且 lookup 失败优雅降级（仍创建 trace，
   只是 EXPERIMENT_DATASET_ID OTel attr 不写，evaluator filter `dataset any_of` 不命中）。
6. **dataset_name=None 时跳过 run_item 关联，不抛错**：保持回退路径合法。
7. **smoke 用 admin role token（临时签）**：评测路径与 admin/user 鉴权差异无关。
