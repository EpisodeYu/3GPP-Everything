## 4. 实施顺序（与 `06-...md §0` 子里程碑表对齐）

按 M7.0 → M7.6 顺序，每段门禁全绿才进下一段。下面只列每段的"先做什么 / 文件落点 / 自测命令"，详细验收清单见 `06-...md §12`。

### 4.1 M7.0 金标准 v1 → v1.5（人 + agent 协作）

**先做什么**：

1. Agent 落 `eval.cli golden validate` + `eval.cli golden merge` 两个子命令（参考 `eval/cli.py` 现有 typer 结构）
2. **人** 用 `eval/golden/_template.yaml` 写题，至少 20 题，重点补 negative / formula / tool / multi_section 四类
3. 人写题边写边跑 `eval.cli golden validate --file eval/golden/v1.handwritten.yaml` 自检
4. agent 跑 merge → 把 handwritten 合并进 v1.yaml；同步 v1.yaml 顶部 `total` / `categories` / `sources` 字段

**文件落点**：

- `eval/cli.py` — 加 `golden` typer app（validate / merge）
- `eval/golden/validator.py` — pydantic-style validator（必填 / 枚举 / id 唯一）
- `eval/tests/unit/test_golden_validator.py` — 单测
- `eval/golden/v1.handwritten.yaml` — 人创建
- `eval/golden/v1.yaml` — agent 合并写入

**自测命令**：

```bash
cd eval && uv run python -m eval.cli golden validate --file eval/golden/v1.handwritten.yaml
cd eval && uv run python -m eval.cli golden merge \
    --base eval/golden/v1.yaml \
    --add eval/golden/v1.handwritten.yaml \
    --out eval/golden/v1.yaml
cd eval && uv run pytest tests/unit/test_golden_validator.py -v
```

**门禁**：

- v1.yaml 题数 ≥ 140
- 分布按 §3.4 容差 ±5：def ~30 / proc ~35 / multi ~10 / table ~10 / formula ~10 / tool ~10 / neg ~15
- [human] 至少 20 题（手写部分）由懂 3GPP 的人 review

### 4.2 M7.1 端到端 runner + 第一档阈值

**先做什么**：

1. 新建 `eval/runner.py`：
   - `run_eval(golden_path, source_filter=None, subset=None) -> list[EvalResult]`
   - HTTP 客户端：复用 `eval/settings.py::EvalSettings`；走 `POST /api/v1/sessions/{sid}/messages` 取 SSE
   - SSE 解析：复用 backend `tests/integration/api/test_chat.py` 的事件 parser 思路（10 类 event）
   - metrics：`fact_coverage`（substring）/ `forbidden_violations` / `must_say_not_found_passed` / `context_recall_section` / `context_recall_spec`（复用 `eval/retrieval/metrics.py::compute_metrics`）
2. 输出：`eval-results/{ts}/{report.md, results.json}`，结构对齐 §4
3. CLI 入口：`python -m eval.runner --golden ... [--source hand_crafted] [--subset N]`
4. `backend/tests/eval/test_golden_v1.py`：写 `test_golden_v1_daily`（D13 宽松断言）+ `test_golden_v1_full`（暂用宽松，M8 收紧）
5. Makefile 加 `eval-daily` / `eval-weekly` 两条

**文件落点**：

- `eval/runner.py` — 新增
- `eval/sse_parser.py` — 新增（提取自 backend test_chat 的 parser，避免循环依赖）
- `eval/tests/unit/test_runner.py` — mock SSE 流的 metrics 单测
- `backend/tests/eval/test_golden_v1.py` — 新增
- `Makefile` — 加 `eval-daily` / `eval-weekly` target
- `docs/03-development/06-...md §4` — 文档对齐

**自测命令**：

```bash
# unit
cd eval && uv run pytest tests/unit/test_runner.py -v
# integration（需起 backend + bootstrap-admin）
cd eval && uv run python -m eval.runner --source hand_crafted --subset 5  # smoke
```

**门禁**：

- runner unit 全绿；daily 子集 < 10min；负样本 100% 过 `must_say_not_found_passed`

## 8. 风险与边界

| 风险 | 缓解 |
|---|---|
| 手写题主观偏差（一人写一人审） | M7.0 [human] 至少 20 题人审；§13 风险表已记 |
| Daily eval 暴露 retrieval 质量问题（如 proc-005） | M7.5 数据 drive 调参；不预设方向 |
| Langfuse Cloud 国内访问抖动 | runner 缺 key / 网络挂时 disable，不阻塞主流程 |
| ApiUsage 写入引入 LLM 调用延迟 | hook 在 response 后异步入库（fire-and-forget）；不阻塞 chat |
| Daily CI cron 漂移到 LLM 限流时段 | mimo RPM=100、TPM=10M 充足；20 题 × 8 call 远低于阈值 |
