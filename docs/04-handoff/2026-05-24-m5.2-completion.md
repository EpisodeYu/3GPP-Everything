### SSE 解析（§8 对齐）

- `SseLineParser.feed(line)`：按 SSE spec 处理 `:` comment / `event:` / 多行 `data:` 拼接 / `id:` / `retry:` 忽略 / 冒号后单空格剥离
- 空行 → emit 一帧 `SseFrame{event, data}`
- `sseFramesFromBytes(Stream<List<int>>)`：utf8.decode → LineSplitter → 喂解析器；正确处理跨 chunk 边界（单测验证）
- 测试覆盖了 sse-starlette 每 15s `: ping` 注释行 + dio responseType=stream 字节流被任意切片这两个生产期场景

### 10 类 ChatEvent sealed-style

```dart
sealed class ChatEvent {
  static ChatEvent fromFrame(SseFrame frame) { ... }
}

class RunStartEvent  { runId, sessionId, messageId }
class NodeStartEvent { node }
class NodeEndEvent   { node, durationMs, summary: Map<String,dynamic> }
class ChunksHitEvent { chunks: List<ChunkPreview> }
class ChunksRerankEvent { chunks: List<ChunkPreview> }   // 与 hit 同序但带 rerank_score；UI 覆盖 hit
class TokenEvent     { delta }
class FinalEvent     { messageId, answer, citations: List<Citation>, confidence }
class CancelledEvent { reason }
class ErrorEvent     { code, message }
class EndEvent       { }
class UnknownChatEvent { name, data }                    // 后端扩展 event name 时降级
```

未知 event 名 → `UnknownChatEvent`，UI 忽略，不让状态机崩。后端字段日后扩展不破坏前端。

### ChatController 状态机

`AutoDisposeFamilyAsyncNotifier<SessionChatState, String /*sid*/>`：

- `build(sid)`：调 `GET /sessions/{sid}/messages` 拉历史；失败时不抛错而是返回空 history 让 composer 仍可用
- `send(content, mode, explicitTools)`：跑 SSE 流；事件驱动更新 partialAnswer / nodes / chunksHit / chunksRerank / citations / confidence；终态 done / cancelled 会 flush 到 history，error 保留在 run 上让 UI 展示
- `cancel()`：先发 `DELETE /sessions/{sid}/runs/{rid}`，再 cancel dio CancelToken 兜底
- `RunStatus`：idle / streaming / cancelling / done / cancelled / error；`isRunning` 仅 streaming / cancelling 为 true
- 运行中再次 send 是 no-op；防止前端用户重复点导致两条 SSE 流并发

### UI 接入要点

- `_ChatView`（chat_page.dart）顶部 `_Header` 展示标题 / mode / status；中间 `_MessagesList` 同时渲染 history + 当前 streaming 的临时 user+assistant 气泡 + 流式累积的 `StreamingAssistantBubble`（partial 为空时显示 typing dots）；底部 `NodeStatusStrip` + 错误 banner（status=error 时显示）+ `Composer`
- `archived_branch` 会话：不渲染 Composer，显示只读 banner（不阻挡历史滚动）
- `ref.listen` 监听 chat state 变化 → `_scrollToBottom()`（postFrame 滚到底部）
- LaTeX：块级 `$$…$$` 用 `flutter_math_fork`；内联 `$…$` M5.3+ 视语料反馈再加（避免与 markdown 美元符号冲突）

## 自主决策记录（CLAUDE.md §4.3）

1. **不引 freezed / build_runner**：M5.0 / M5.1 都没引；本期 10 类 SSE event 用 sealed-style 抽象类 + 子类 + 静态 `fromFrame` 工厂手写完成（约 200 行），可读 + IDE 跳转友好；后端日后加新 event name 走 `UnknownChatEvent` 降级，不强结构化 chunk_id 之外的字段。如果 M5.4 起 checkpoint UX / fork / rollback 让 union 类型膨胀到 20+，再一次性开 freezed。
2. **手写 SSE 解析器 + 暴露 `parseSseLines(Iterable<String>)`**：测试用同步喂行接口，避免每个 case 都做 utf8 字节流构造；生产用 `sseFramesFromBytes(Stream<List<int>>)` 通过 LineSplitter 解决跨 chunk 字节边界（单测验证）。冒号后单空格按 W3C SSE spec 严格剥掉一次，保留剩余空格。
3. **`StreamingAssistantBubble` 单独 widget**：流式累积的 partial answer 不进 history（history 只放固化后的 MessageOut），单独走一个临时气泡渲染；token 累积期间空字符串显示 typing dots 让用户知道 backend 正在跑（特别是 retrieve 节点 5-10s 没 token 输出时）。
4. **`_flushDoneToHistory` 只在 done / cancelled 时固化**：error 留在 run 上让 UI 展示错误 banner + 用户能立刻重发；如果 error 也固化进 history，会让历史里出现"失败" 状态的占位气泡，体验更差。
5. **cancel 双通道**：先发 `DELETE /runs/{rid}`（后端 race loop 命中可在 LLM streaming 中段就停）；再 cancel dio CancelToken（断连兜底）。后端口径 `_iter_with_cancel` 的 race-cancel-event 通道与本前端实现完全对齐。
6. **`AutoDisposeFamilyAsyncNotifier<SessionChatState, sid>`**：每个会话一份状态，切会话时 autoDispose 析构正在跑的 stream。单测里发现 `c.read()` 单次访问会让 autoDispose 立刻析构 → 用 `c.listen` 保活，仿 widget mount 行为；后续 chat_controller 测试都用同样的 helper（`_keepAlive`）。
7. **Composer Shift+Enter 换行**：用 `Focus.onKeyEvent` 拦截 Enter（无 Shift → 触发 send，吞事件）；有 Shift → ignore，让 TextField 原生处理换行。
8. **不接 `Last-Event-ID` 自动重连**：docs §8 明说 "MVP 简单不重连，断了让用户重发"；本期保留这个简化，未把 reconnection 列入门禁。
9. **`chunks_hit` / `chunks_rerank` 仅落 state 不开 ChunksPanel UI**：docs §5.3 chunks 预览抽屉 + 卡片 + 跳阅读器是 M5.3 的范围（依赖 Reader 页面）。本期把数据通进 state，UI 留给 M5.3 拓展，避免 M5.2 越权。
10. **不引 `markdown_widget` / 其他三方 markdown**：沿用 docs §3 已列的 `flutter_markdown_plus`，块级 LaTeX 用 `flutter_math_fork`；如果后续渲染体验有问题（如 table 渲染丑、code 块字号怪），M5.6 整体 polish 时再换。
11. **integration_test 把"完整 send → token → final + cancel"压成一个 case**：原本想拆两个 case（happy path + cancel），但发现第二轮 pumpWidget 重建后 chat_controller 重新走 `build()` 加历史；时序不稳。压成一个 case：`send → run_start → token → token → 用户中途取消 → cancelled → end`，既覆盖 token 累积、又覆盖 cancel + cancelRun + flush，状态机分支比"全跑到 final"更广。

## 剩余风险 / 已知问题

- **真"有窗口"chrome 仍不在本机覆盖范围**：同 M5.1，本机 Linux 无 X server / 无 sudo；headless chrome 已是 CI 等价物，docs §14 的 `[human]` 仍需人在 macOS / Windows 真带窗口浏览器上手验收 UX。
- **stream_closed 在生产期可能误报**：如果 nginx / 中间代理在 15s `: ping` 之间断流，前端会标 error。`receiveTimeout: Duration.zero` 已经放开了 dio 30s 默认；生产 nginx 还需要 `proxy_buffering off` + `X-Accel-Buffering: no`（docs §13 已记，M5.6 部署时确认）。
- **partialAnswer 在 cancel 后仍固化进 history**：用户可能预期"取消后内容消失"，但本实现保留 partial（带"已取消"角标）；产品上更合理（用户能看到已生成的部分作上下文）。如果 UX 反馈倾向丢弃，改 `_flushDoneToHistory` 的 cancelled 分支：content = ''。
- **`UnknownChatEvent` 不警告**：后端日后加新 event name，前端静默忽略；如果加了关键事件（如 paused for checkpoint），UI 没接会让用户卡。M5.4 接 checkpoint 时把 paused / resumed 加进 sealed-style。
- **节点 summary bottom sheet 关掉只能点空白**：M5.4 / M5.6 再 polish 加 ESC / 头部 X 按钮。
