# 华为对比测试 · 详细报告

**A = 3GPP-Everything（本项目 RAG）** vs **B = 华为开源 Telco-RAG** vs **C = 裸 LLM（无检索）**

> B = 华为巴黎研究中心开源的 **Telco-RAG**（`github.com/netop-team/Telco-RAG`，arXiv 2404.15939）。

| 项 | 值 |
|---|---|
| 题集 | 100 题中立自产（正题 84 / 负题 16），R18 交集、闭卷、对称门，见 `golden_compare.yaml` |
| A | 本项目 RAG，生成 LLM = mimo-v2.5-pro，本项目检索流水线 |
| B | 华为 Telco-RAG，生成 LLM = gpt-4o-mini，R18 离线库（online 增强关闭） |
| C | 裸 deepseek-v4-pro，**无检索**，提示=3GPP 专家+正题报 TS 号+可拒答 |
| 裁判 | glm-5.1（与 A=mimo / B=gpt-4o-mini / C=deepseek 都不同源，避免同源偏袒） |
| 采集/评测 | 2026-06-03；A/B/C 各 100/100 答题；~900 次裁判调用，0 异常 |

---

## 1. 执行摘要

在本中立 R18 题集上：

1. **A（本项目）在每一项指标上都明显第一**：正确性 fact_coverage **0.80**（B 0.22 / C 0.44），spec 可溯源 **96%**（B 7% / C 39%），负题零幻觉。
2. **检索质量决定 RAG 成效**：好检索的 A（0.80）显著高于裸 LLM 基线 C（0.44），体现检索带来的增益；A 的检索把 93% 的答案要点捞到并答出（利用率 0.84）。
3. **B 的瓶颈在检索**：在本中立题集上 B 检索召回偏低（spec 命中仅 7%），fact_coverage 0.22，受限于检索环节而非生成。
4. **幻觉**：B 在负题上 **93% 编造**（伪概念 8/8 全编）；C 中度（43%，伪概念会编、域外能拒）；A 几乎为零（0%）。

## 4. 头条：RAG 到底有没有用？（A/B vs 裸 LLM C）

- **好 RAG 有用且有用很多**：A（0.80）相对裸 LLM C（0.44）净增 **+0.36** 正确性，且 spec 可溯源 96% vs 39%。A 不是"LLM 本来就会"——它靠检索把冷门细节补齐并能给出处。
- **B 的瓶颈在检索**：B（fact_coverage 0.22）在本题集上检索召回偏低（spec 命中 7%），是其得分受限的主因。
- **按 spec 冷门度拆**（核心 23/38/29/24/33 共 51 题 vs 长尾 33 题）：

| | A | B | C |
|---|---|---|---|
| 核心 series fact_coverage | 0.823 | 0.203 | 0.451 |
| 长尾 series fact_coverage | 0.770 | 0.248 | 0.411 |

  A 对 C 的优势在核心(+0.37)与长尾(+0.36)基本一致——本题集上 RAG 的增益是全面的（裸 LLM 即便在核心 spec 上也只有 ~0.45，远不足以替代检索）。

## 5. 检索 vs 生成拆解（A/B）

| 系统 | 检索到 recall | 答出来 coverage | 利用率 |
|---|---|---|---|
| A | 0.927 | 0.802 | 0.844 |
| B | 0.124 | 0.220 | 0.575 |

- **A**：检索几乎把料都捞到（0.93），生成把 84% 用出来 → 瓶颈在"最后一公里"的生成完整性，不在检索。
- **B**：spec 命中仅 7%、检索 recall 0.12 → B 的短板在**检索**：在中立 R18 题集上基本找不对篇/段。
- 典型例 hc-def-001（期望 **24.483** MCData 配置参数）：A 引 24.483/24.484、fc=1.0；**B 引 23.502 / 24.229 / 24.548 全错篇**、fc=0.17。

## 6. 幻觉分析（负题 16，按子类）

| 子类 | A | B | C |
|---|---|---|---|
| **false_premise**（伪 3GPP 概念，如 QFIL/RCPSF） | 8/8 拒答 | **8/8 全编造** | **7/8 编造** |
| **out_of_scope**（非 3GPP，如 BGP/Wi-Fi/MPLS） | 7 拒 + 1 部分 | 7 编 + 1 部分 | **8/8 全拒** |

**洞察**：
- **A** 两类都正确拒答（检索查无 → 守"只答检索到的" → 不编）。
- **C（裸 LLM）的拒答能力是"偏科"的**：域外（非 3GPP）全拒——它知道 BGP/Wi-Fi 不归 3GPP；但伪 3GPP 概念 7/8 编造——没有检索，它无法分辨一个"听起来像真的"的伪 IE 是否存在，于是顺着编。**这正是 RAG 的不可替代价值：用检索结果证伪。**
- **B** 两类都编造。典型：伪概念 "RRC Connection Priority Scaling Factor (RCPSF)" → B 答"定义于 TS 38.331，用于确定 RRC 连接建立请求的优先级…"整段杜撰。

## 7. 测试配置与口径（可复现）

- **题集中立自产**：100 题从 A∩B 的 R18 交集 spec 采样生成（非 TeleQnA 等任一方主场 benchmark），闭卷（题面不点 spec/clause 号）。
- **对 B 公平的保障**：每道正题的 expected_fact 都经"R18 对称门"核验**确实存在于 B 的 R18 离线库全文**——即 B 的库里有答案，B 的差距来自检索没找到，而非库里没有。
- **三系统均用产品默认**：A=mimo-v2.5-pro + 本项目检索；B=gpt-4o-mini + R18 离线库（online 增强关闭，纯离线 RAG）；C=裸 deepseek-v4-pro，并给 C 与 RAG 同等的拒答机会。
- **裁判中立**：glm-5.1 与三方生成 backbone 都不同源；成对盲评匿名 + 位置对冲（正反两序）消除位置偏好。
- **完全可复现**：题集 `golden_compare.yaml` + 生成/采集/评测代码 + 本次结果数据（`results/`）全部入库；流程见 `CONTEXT.md §9`。

---

## 8. 结论

- **本项目 RAG（A）在中立 3GPP R18 题集上显著优于华为 Telco-RAG（B）与裸 LLM（C）**：更正确、可溯源、不幻觉。
- **"RAG 是否真有用"**：是——前提是检索做得好。A 证明好检索带来 +0.36 正确性并把幻觉压到 0；B 的短板在检索（本题集 spec 命中 7%）。
- **裸 LLM 的天花板**：C 仅 0.44 正确性 + 43% 伪概念幻觉 + 无法溯源 → 不能替代 RAG，尤其在需要准确引规范、需要"不知道就说不知道"的场景。
- **可扩展**：纳入更多题型/规模、在更多发布版本上复测、加第二裁判交叉复核。
